MODULE 6

Web Application Security

Introduction

Web applications face unique security challenges due to their exposure to the internet and complexity. Understanding common vulnerabilities, secure coding practices, and defense mechanisms is essential for protecting web applications from compromise. The OWASP Top 10 provides a comprehensive framework for understanding the most critical web application security risks.

Critical: Validate All Input
The most important web security principle: Never trust user input. Validate, sanitize, and encode all data from untrusted sources (HTTP requests, cookies, headers, etc.). Almost every web vulnerability exploits unvalidated or improperly handled user input.

OWASP Top 10:2021

OWASP (Open Web Application Security Project) maintains the OWASP Top 10, a standard awareness document representing the most critical web application security risks. Updated regularly to reflect evolving threats.

Rank Category Description
A01 Broken Access Control Improper restrictions on authenticated users, IDOR, privilege escalation
A02 Cryptographic Failures Sensitive data exposure, weak encryption, cleartext transmission
A03 Injection SQL injection, NoSQL injection, command injection, LDAP injection
A04 Insecure Design Flawed architecture, missing security controls, business logic errors
A05 Security Misconfiguration Default accounts, unnecessary features, verbose error messages

A08 Software and Data Integrity Failures Integrity verification issues, insecure deserialization, CI/CD pipeline attacks

SQL Injection (SQLi)

SQL injection occurs when attacker-supplied data modifies database query structure. SQLi is one of the most common and dangerous web vulnerabilities, allowing attackers to read, modify, or delete database data, bypass authentication, or execute operating system commands.

How SQL Injection Works

When input is concatenated directly into SQL query string, attacker can manipulate query structure:

Vulnerable Code (PHP)
$username = $_POST['username']; $password = $_POST['password']; // VULNERABLE: Direct concatenation $query = "SELECT * FROM users WHERE username = '$username' AND password = '$password'"; $result = mysqli_query($conn, $query); Normal login:
username: alice, password: password123
Query: SELECT * FROM users WHERE username = 'alice' AND password = 'password123'
Result: Returns alice's row if password correct Attacker input:
username: ' OR '1'='1
password: (anything)
Query: SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...'
Result: Bypasses authentication ('1'='1' always true, ignores actual username)

Types of SQL Injection

Union-Based SQL Injection

Attacker injects UNION statement to combine results from injected query with original query:

SELECT * FROM products WHERE category = 'foo' UNION SELECT username, password FROM users --'

Attacker can enumerate database structure and extract sensitive data by matching column counts.

Boolean-Based Blind SQL Injection

Attacker exploits application behavior to deduce information:

SELECT * FROM users WHERE username = 'admin' AND 1=1 --' // True SELECT * FROM users WHERE username = 'admin' AND 1=2 --' // False

Attacker observes response differences (success/failure, page changes) to extract data one bit at a time.

Error-Based SQL Injection

Attacker triggers database errors to reveal information about database structure:

' AND 1=CONVERT(int, (SELECT TOP 1 table_name FROM information_schema.tables)) --'

Database error message may contain table name, revealing schema information.

SQL Injection Prevention

Primary Prevention: Parameterized Queries
Use parameterized queries (prepared statements) - database treats parameters as data, not executable code: // SECURE: Parameterized query (Python/Psycopg2) cursor.execute( "SELECT * FROM users WHERE username = %s AND password = %s", (username, password) ) // SECURE: Parameterized query (PHP/PDO) $stmt = $pdo->prepare("SELECT * FROM users WHERE username = ? AND password = ?"); $stmt->execute([$username, $password]); Parameterized queries completely prevent SQL injection when used correctly.

Additional SQLi Defenses

  • ORM (Object-Relational Mapping): Use frameworks that handle SQL generation and parameterization automatically (e.g., Django ORM, Sequelize)
  • Input validation: Validate input format, type, and length before processing (but never rely on validation alone - always use parameterized queries)
  • Least privilege database accounts: Application should not use admin-level database account (no DROP TABLE, no schema modifications)
  • Stored procedures: Well-designed stored procedures can help, but must be carefully implemented to avoid SQLi within procedures themselves
  • Web Application Firewall (WAF): Can block known SQLi patterns, but WAFs can be bypassed (defense in depth, not primary defense)
Common SQLi Myths
Myth: "I escape quotes, so I'm safe"
Reality: Quote escaping is error-prone and may not account for encoding or database-specific escaping. Use parameterized queries. Myth: "I validate input, so SQL injection impossible"
Reality: Input validation alone insufficient. Always use parameterized queries regardless of validation. Myth: "I use stored procedures, so I'm immune"
Reality: Stored procedures vulnerable if they concatenate input internally. Parameterized stored procedures required.

Cross-Site Scripting (XSS)

XSS occurs when attacker-supplied data is reflected or stored in web pages and executed by victim browsers. XSS allows attackers to steal session cookies, modify page content, redirect users, or execute arbitrary JavaScript in victim's browser context.

XSS Types

Stored XSS (Persistent XSS)

Malicious script stored on server and executed when victims access affected page:

  • Example: Attacker posts comment containing JavaScript on forum
  • Vulnerable site: Displays comments unsanitized to all visitors
  • Effect: All visitors who view comment execute JavaScript in their browser
  • Severity: High - affects many users silently

Reflected XSS

Malicious script reflected back to victim without storage:

  • Example: Search parameter displayed unsanitized: search?q=<script>evil()</script>
  • Vulnerable site: Returns page showing "Search for: <script>evil()</script>"
  • Attack: Attacker sends phishing link: https://site.com/search?q=<script>evil()</script>
  • Effect: Victims who click link execute script
  • Severity: Medium - requires victim interaction (clicking link)

DOM-Based XSS

Vulnerability in client-side JavaScript (data flow entirely in browser):

  • Example: URL hash processed and inserted into DOM via innerHTML
  • URL: https://site.com/#<script>evil()</script>
  • Vulnerable code: document.getElementById('output').innerHTML = location.hash
  • Effect: Script executes without server involvement
  • Severity: Medium to High - bypasses some server-side filters

XSS Attacks

XSS enables attackers to perform many attacks within victim's browser context:

  • Session hijacking: Steal authentication cookies and impersonate victim
  • Keylogging: Capture keystrokes and transmit to attacker server
  • Phishing modifications: Modify page content to create convincing phishing site
  • Redirection: Redirect victim to malicious site
  • CSRF payload delivery: Use XSS to perform CSRF attacks (see later section)
  • DOM manipulation: Modify page behavior, insert malicious forms

XSS Prevention

Primary Prevention: Output Encoding
Always encode data before outputting into HTML context. Use appropriate encoding for context: // HTML entity encoding (for content) < & > // HTML attribute encoding (for attributes) < & > " ' // JavaScript encoding (for scripts) \x3C\u003C ... // URL encoding (for URLs) %3C ... Use framework templating engines with automatic encoding (e.g., Jinja2, React JSX, Vue templates)
// SECURE: Framework with auto-encoding (Jinja2/Python) template = render_template('page.html', user_input=request.form['data']) // SECURE: React (auto-sXSS protection) const Component = ({ data }) => <div>{data}</div> // MANUAL: HTML entity encoding (if framework unavailable) function encodeHTML(str) { return str.replace(/&/g, '&') .replace(/</g, '<') .replace(/>/g, '>') .replace(/"/g, '"') .replace(/'/g, '''); }

Content Security Policy (CSP)

CSP is HTTP header that restricts sources of scripts, styles, images, and other resources. CSP prevents XSS by disallowing inline scripts and restricting script sources.

# Content-Security-Policy header example Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted.cdn.com; style-src 'self' 'unsafe-inline'; img-src 'self' data:; object-src 'none'; frame-ancestors 'none';
  • script-src: Restricts script sources (disallows inline scripts by default)
  • default-src 'none';: Disallows loading resources from anywhere unless explicitly allowed
  • frame-ancestors 'none';: Prevents clickjacking
  • object-src 'none';: Disallows plugins (Flash, Java)
CSP Report-Only Mode
Before enforcing CSP, use report-only mode to observe violations without blocking: Content-Security-Policy-Report-Only: default-src 'self'; ... Monitor CSP violation reports to understand site dependencies before enforcing strict policy.

XSS Cookies Defense

// Secure cookie flags Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Strict
  • HttpOnly: Cookie not accessible via JavaScript, preventing session theft via XSS
  • Secure: Cookie transmitted only over HTTPS, not HTTP
  • SameSite=Strict / SameSite=Lax: CSRF protection (see next section)

Cross-Site Request Forgery (CSRF)

CSRF tricks victim's browser into performing unwanted actions on site where victim is authenticated. Attacker doesn't need victim's credentials - victim's browser automatically sends authentication cookies with requests.

How CSRF Works

When user logs into site, site sets authentication cookie. While logged in, browser sends cookie with every request to that site automatically.

CSRF Attack Example
Victim is logged into bank.com (authentication cookie set) Attacker creates malicious site: evil.com/attack.html <body onload="document.forms[0].submit()"> <form action="https://bank.com/transfer" method="POST"> <input name="to" value="attacker_account"> <input name="amount" value="10000"> </form> Victim visits evil.com. Browser: 1. Automatically submits form to bank.com 2. Includes victim's authentication cookie (sent automatically) 3. Bank processes transfer, believing request from authenticated victim 4. Money transferred to attacker account Victim did not intend this action - browser simply followed attacker's instructions.

CSRF Prevention

Primary Prevention: CSRF Tokens
Anti-CSRF tokens are random, unpredictable tokens embedded in forms (and validated on submission): Flow: 1. Server generates random token, stores in session 2. Server embeds token in HTML forms (or state-changing requests) 3. Client submits form with token 4. Server validates token matches session token 5. If valid, request processed; if invalid or missing, rejected Why it works: Attacker cannot guess or read token (same-origin policy prevents cross-site JavaScript from reading token), so attacker cannot construct valid request.
// Server-side (Python/Flask example) from flask_wtf.csrf import CSRFProtect csrf = CSRFProtect(app) # Template (auto-injects CSRF token) <form method="POST"> <input type="hidden" name="csrf_token" value="{{ csrf_token() }}"> <!-- other form fields --> </form>

SameSite Cookie Attribute

SameSite attribute controls whether cookies sent with cross-site requests.

SameSite Values
SameSite=Strict: Cookie sent only for same-site requests (first-party site) SameSite=Lax: Cookie sent for same-site requests and safe cross-site requests (top-level navigation, GET requests) SameSite=None: Cookie sent for all requests (must be combined with Secure attribute for HTTPS)
// Recommended default: SameSite=Lax Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Lax // For scenarios requiring cross-site login, SameSite=None Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=None
CSRF Defense in Depth
Combine CSRF defenses: 1. CSRF tokens for state-changing requests 2. SameSite=Lax cookies (blocks most CSRF, allows legitimate navigation) 3. Verify HTTP Origin and Referer headers where possible 4. Requiring double-submit cookie (less common, but additional layer) 5. Custom HTTP headers (X-Requested-With: XMLHttpRequest)

Authentication and Session Management

Broken authentication allows attackers to compromise user accounts, impersonate legitimate users, or bypass authentication completely. Strong authentication and secure session management are critical.

Password Storage

Passwords must never be stored in plaintext or using fast hashes. Always use password hashing algorithms.

// SECURE: Argon2id (recommended) import argon2 hasher = argon2.PasswordHasher(time_cost=2, memory_cost=102400, parallelism=8) hash = hasher.hash("user_password") hasher.verify(hash, "user_password") // SECURE: bcrypt (widely supported) import bcrypt salt = bcrypt.gensalt() hash = bcrypt.hashpw("user_password".encode(), salt) bcrypt.checkpw("user_password".encode(), hash) // SECURE: PBKDF2 (when Argon2/bcrypt unavailable) import hashlib, os, binascii def password_hash(password): salt = os.urandom(32) key = hashlib.pbkdf2_hmac('sha256', password.encode(), salt, 100000) return binascii.hexlify(salt + key).decode() def password_verify(password, stored): salt = binascii.unhexlify(stored[:64]) key = binascii.unhexlify(stored[64:]) candidate = hashlib.pbkdf2_hmac('sha256', password.encode(), salt, 100000) return key == candidate
Critical: Never Use Plaintext or Fast Hashes
Do NOT store passwords: - In plaintext - Using MD5, SHA-1, SHA-256 (too fast - attacker can hash billions per second on GPU) - Without salt (rainbow tables enable mass cracking) Always use Argon2id, bcrypt, or PBKDF2 (prefer Argon2id).

Multi-Factor Authentication (MFA)

MFA requires multiple authentication factors, dramatically reducing account compromise risk.

Authentication Factors
Something you know: Password, PIN, security questions Something you have: Smartphone, hardware token, smart card Something you are: Biometrics (fingerprint, face recognition) Example: Password (something you know) + TOTP code from authenticator app (something you have) = 2FA
  • TOTP (Time-based One-Time Password): Google Authenticator, Authy - codes change every 30 seconds based on shared secret
  • SMS-based codes: Less secure (SMS interception possible), but better than no MFA
  • Hardware tokens: YubiKey, hardware security keys - strongest MFA
  • Push notifications: Duo, Microsoft Authenticator - user approves login attempt

Session Management

Secure Session Management Practices

  • Regenerate session ID after login: Prevent session fixation attacks
  • Set appropriate session expiration: Shorter inactivity timeout, maximum lifetime
  • Secure session cookies: HttpOnly, Secure, SameSite (see XSS section)
  • Destroy session on logout: Invalidate server-side session and session cookie
  • Bind session to additional attributes: IP address, user-agent (detect session hijacking)
  • Concurrent session limits: Limit number of simultaneous sessions per user
// Flask session example (server-side secure sessions) from flask import Flask, session app = Flask(__name__) app.secret_key = 'strong_random_secret_key' @app.route('/login') def login(): session.clear() # Clear any existing session before login # ... verify credentials session['user_id'] = user.id session.regenerate() # Regenerate session ID @app.route('/logout') def logout(): session.clear() # Destroy session

Security Misconfiguration

Security misconfiguration is the most common vulnerability category, encompassing default configurations, unnecessary features enabled, outdated software, and excessive information disclosure.

Common Security Misconfigurations

1. Default Credentials

Default admin accounts with default passwords (admin/admin, root/toor, scott/tiger)

Prevention: Change all default credentials during initial setup. Remove default accounts.

2. Verbose Error Messages

Error messages revealing stack traces, database errors, file paths

Example: "Error: Table 'users' doesn't exist" reveals database schema

Prevention: Display generic error messages to users, log detailed errors server-side

3. Directory Listing Enabled

Web server automatically lists directory contents when index file missing

Prevention: Disable directory listing globally, ensure all directories have index files

4. Unnecessary Features Enabled

Debug mode, test endpoints, unused ports, demonstration pages

Prevention: Disable debug mode in production, remove test endpoints, firewall unused ports

5. Exposed Sensitive Files

.git directory, .env files, backup files, configuration with secrets

Prevention: Block access to sensitive files, ensure .htaccess or web.config prevents access

6. HTTP Headers Misconfigured

Missing security headers, CORS overly permissive

Prevention: Implement proper security headers

Security Headers

# Essential security headers Content-Security-Policy: default-src 'self'; script-src 'self' X-Content-Type-Options: nosniff X-Frame-Options: DENY X-XSS-Protection: 1; mode=block Strict-Transport-Security: max-age=31536000; includeSubDomains Referrer-Policy: strict-origin-when-cross-origin Permissions-Policy: geolocation=(), microphone=(), camera=()
  • X-Frame-Options: Prevents clickjacking (DENY or SAMEORIGIN)
  • HSTS: Forces HTTPS connections, prevents downgrade attacks
  • X-Content-Type-Options: Prevents MIME-sniffing
  • Permissions-Policy: Disables sensitive browser features (geolocation, camera)

Server Configuration Hardening

# Nginx security configuration server { listen 443 ssl http2; server_name example.com; # SSL configuration ssl_certificate /path/to/cert.pem; ssl_certificate_key /path/to/key.pem; ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers HIGH:!aNULL:!MD5; # Security headers add_header X-Frame-Options "SAMEORIGIN" always; add_header X-Content-Type-Options "nosniff" always; add_header X-XSS-Protection "1; mode=block" always; # Disable server tokens (hide nginx version) server_tokens off; # Disable directory listing autoindex off; # Block access to sensitive files location ~ /\. { deny all; } }

Server-Side Request Forgery (SSRF)

SSRF occurs when application fetches remote resource without validating URL, allowing attacker to force server to make requests to arbitrary destinations. SSRF can access internal services, scan internal networks, or read sensitive data.

SSRF Examples

SSRF via Webhook URL
Application allows users to specify webhook URL for notifications. Vulnerable code: url = request.POST['webhook_url'] requests.get(url) Normal usage: https://user.example.com/webhook (legitimate external URL) Attacker input: http://localhost:8080/admin/delete_user (internal admin endpoint) Server makes HTTP request to localhost:8080/admin/delete_user, which may be internal admin API. Attacker bypasses firewall because request originates from trusted server.

SSRF Attacks

  • Internal port scanning: http://localhost:80, http://localhost:8080, http://localhost:3306
  • Access internal services: http://10.0.0.1/admin, http://192.168.1.1/config
  • File access: file:///etc/passwd (if protocol whitelist not strict)
  • Cloud metadata access: http://169.254.169.254/latest/meta-data/ (AWS cloud metadata)

SSRF Prevention

  • URL whitelisting: Allow only specific domains or IP ranges
  • Network segmentation: Run application in network zone with restricted outbound access
  • Validate hostnames: DNS rebinding protection, resolve hostname and validate before request
  • Disable dangerous schemas: Block file://, gopher://, dict:// and non-HTTP protocols
  • Use safe libraries: Configure HTTP libraries to disallow redirects to internal addresses
# SSRF prevention example (Python) import urllib.parse import ipaddress ALLOWED_DOMAINS = ['api.example.com'] def is_valid_url(url): parsed = urllib.parse.urlparse(url) # Protocol whitelist if parsed.scheme not in ['http', 'https']: return False # Domain whitelist hostname = parsed.hostname if hostname not in ALLOWED_DOMAINS: return False # DNS resolution validation (prevent private IPs via DNS rebinding) try: ip = ipaddress.ip_address(socket.gethostbyname(hostname)) if ip.is_private or ip.is_loopback: return False except: return False return True # Usage if is_valid_url(user_supplied_url): requests.get(user_supplied_url) else: raise ValueError("Invalid URL")

Secure Development Lifecycle

Secure development integrates security throughout software lifecycle, from design to deployment. Shift-left security reduces cost and impact of vulnerabilities.

Secure Coding Practices

  • Never trust user input: Validate, sanitize, encode all input
  • Principle of least privilege: Applications run with minimal privileges
  • Defense in depth: Multiple layers of security (not relying on single control)
  • Fail securely: When code fails, fail closed (secure by default)
  • Use secure libraries: Libraries maintained for security, actively patched
  • Keep dependencies updated: Regularly address known vulnerabilities
  • Minimal attack surface: Disable unused features, expose minimal endpoints

Security Testing

Static Application Security Testing (SAST)

Analyze source code for vulnerabilities without executing:

  • Tools: SonarQube, Checkmarx, Semgrep, Bandit (Python), ESLint with security plugins
  • Detects: SQL injection patterns, XSS vulnerabilities, insecure dependencies
  • Benefits: Early detection, integrates with CI/CD
  • Limitations: False positives, cannot catch all vulnerabilities

Dynamic Application Security Testing (DAST)

Analyze running application for vulnerabilities:

  • Tools: OWASP ZAP, Burp Suite, Nessus, Nikto
  • Detects: SQL injection, XSS, misconfiguration, outdated libraries
  • Benefits: Tests actual deployed application, finds runtime issues
  • Limitations: Requires deployed application, may miss business logic flaws

Dependency Scanning

Check dependencies for known vulnerabilities:

# JavaScript (npm) npm audit # Python (pip) pip-audit # Python (safety) safety check # Ruby (bundler) bundle audit # Java (Maven) mvn dependency-check:check

Security Review Checklist

  • Input validation: All user input validated and sanitized
  • Output encoding: All output properly escaped for context (HTML, JavaScript, URL)
  • SQL queries: Parameterized queries only (no concatenation)
  • Authentication: Strong passwords, MFA, secure session management
  • Authorization: Every request validates user has permission for action
  • HTTP headers: Security headers properly configured (CSP, HSTS, X-Frame-Options)
  • TLS/SSL: HTTPS only, strong ciphers, valid certificates
  • Error handling: Generic errors to users, detailed errors logged securely
  • Logging: Security-relevant events logged (login attempts, failures, changes)
  • Dependencies: Updated, no known vulnerabilities
