MODULE 9

Advanced Anonymity

Introduction

Building on basic anonymity tools, advanced anonymity techniques provide stronger protection against sophisticated adversaries. Advanced adversary capabilities include traffic analysis, correlation attacks, and browser fingerprinting. This module covers hardened systems, advanced operational security practices, and specialized tools for maintaining anonymity against capable opponents.

Critical Warning
Advanced anonymity techniques described in this module are for educational purposes and privacy protection. These techniques can also be used maliciously. Responsible use requires ethical boundaries, legal compliance, and awareness that determined adversaries with sufficient resources may eventually overcome any anonymity system. No anonymity solution provides absolute protection against all adversaries, particularly nation-states with access to global surveillance infrastructure.

Deep Web and Dark Web

The deep web encompasses all internet content not indexed by search engines. The dark web is a subset of the deep web intentionally hidden and requiring special software or authorization to access.

Surface, Deep, and Dark Web Distinction

Web Layers
Surface Web (Clear Web): Content indexed by search engines (Google, Bing). Publicly accessible. Deep Web: Content not indexed by search engines. Includes: - Private databases (library catalogs, medical records) - Behind paywalls (news articles, academic journals) - Private accounts (banking portals, webmail) - Internal corporate intranets Most of the web is deep web by volume, not dark. Dark Web: Subset of deep web that is: - Intentionally hidden - Requires anonymity software (Tor, I2P) - Typically hosts both legal and illegal content .onion addresses (Tor) and .i2p (Invisible Internet Protocol) sites

Tor Hidden Services v3 (.onion v3)

Tor hidden services provide server anonymity. Both client and server remain anonymous - server does not know client's IP, client does not know server's IP.

# v3 .onion addresses (56 random characters) # Example: x5x7yz...5wz2.onion (v3 address length) # vs v2 addresses (16 characters) # v3 advantages: # - Improved cryptography (ed25519 instead of RSA) # - Better resistance to enumeration attacks # - No need for directory authority for address discovery # - Longer addresses = more difficult to brute force
  • How it works: Hidden services advertise via hidden service descriptor stored in distributed hash table (DHT) within Tor network
  • Access method: Must use Tor Browser or Tor proxy to access (.onion addresses only resolve through Tor)
  • Anonymity properties: Server IP never revealed, client IP never revealed, Tor relays see only next hop
  • Uses: Secure drop boxes for whistleblowers (SecureDrop, GlobaLeaks), anonymous publishing, anonymous marketplaces
Important Context
.onion sites include both legitimate privacy services (whistleblowing, anonymity tools, privacy-focused communication) and illegal markets. The technology is neutral - it encrypts and anonymizes. User intentions and content vary. Many legitimate services use .onion addresses for enhanced privacy and security.

I2P (Invisible Internet Project)

I2P is anonymous overlay network similar to Tor but designed for different threat model and use cases.

  • Peer-to-peer: Unlike Tor (relay-based), I2P is fully peer-to-peer with distributed infrastructure
  • Garlic routing: Multiple endpoints encrypted together ("garlic") for improved anonymity
  • Tunnels: Unidirectional tunnels (separate inbound and outbound tunnels)
  • Eepsites: .i2p websites (analogous to .onion hidden services)
  • Use cases: File sharing, anonymous email, BitTorrent, I2P-based applications
Tor vs I2P Comparison
Tor: Optimized for web browsing, larger relay network, better for accessing regular internet anonymously, more user-friendly, hidden services (.onion) I2P: Optimized for P2P applications, fully decentralized, better for anonymous file sharing and BitTorrent, eepsites (.i2p), smaller user base Use both: They serve different purposes; I2P for P2P, Tor for web browsing and hidden services

Advanced Tor Configuration and Hardening

Advanced Tor configuration improves anonymity beyond default settings, providing stronger protection against sophisticated adversaries.

Tor Bridges

Bridges are unpublished Tor relays that are not publicly listed. Bridges are used when ISP or firewall blocks connections to Tor directory authorities or public relays.

Bridge Types

  • Plist bridges: Publicly distributed on Tor website (can be blocked by censorship)
  • Private bridges: Distributed privately by email or HTTPS (more resistant to blocking)
  • Obfs4 bridges: Transport obfuscation that disguises Tor traffic as regular HTTPS (resists fingerprinting)
  • Meek bridges: Routes Tor traffic through CDN fronts (like Google App Engine) to blend with regular traffic
# Tor Browser: Configure bridges Tor Browser → Preferences → Tor Network → Use a Bridge # obfs4 bridge example (most common) obfs4 173.255.231.12:443 D7AF0A9912A48E90ED451F0C2D2742E8C9A1C0AF cert=I6yN2vO+mLKE2O2p1lL7E7v8Qq8ZKj0nKtKpKtKpKtKpKpKtKpKtKpKtKpK iat-mode=0 # meek bridge example (highly obfuscated) meek 0.0.2.0:2 MEek 8A4F8C2E8B12F9D8A8C8F8D8E8C8F8D8E8C8F8D8E8C8F8D8E8C8F8D url=https://meek.amazonaws.com/ front=cdn.sstatic.net

Torrc Configuration Hardening

# /etc/tor/torrc (Linux) or Tor Browser torrc # Advanced hardening configurations # Use Entry Guards (long-term guards for first hop) UseEntryGuards 1 NumEntryGuards 3 # Guard lifetime limits (useful if concerned about compromised entry guards) Guardlifetime 90 days # Disable IPv6 (if not needed - reduces fingerprinting surface) IPv6Exit no # DNS and SOCKS proxy (for system-wide use) DNSPort 5353 SocksPort 9050 # Disable exit nodes (use as client only - hidden services only) ExitRelay 0 # Contact info for relay (if operating relay) ContactInfo admin@example.com # Bandwidth limits (if operating relay) RelayBandwidthRate 1 MB RelayBandwidthBurst 2 MB

Tor Browser Security Slider

Tor Browser security slider adjusts between safety and usability.

Security Slider Levels
Safest: JavaScript disabled entirely. Many websites will not work. Maximum protection against browser fingerprinting and exploits. Safer (recommended): JavaScript disabled on non-HTTPS sites and specific dangerous contexts. Good balance of security and usability for most users. Standard: JavaScript enabled but with Tor Browser hardening. Default setting used by most users. JavaScript trade-off: Enabling JavaScript (Standard) increases fingerprinting and exploit risk. Disabling JavaScript (Safest) breaks many websites. Safer disables JavaScript on HTTP sites (which cannot use TLS inspection anyway).

New Identity Misconception

Clicking "New Identity" in Tor Browser changes Tor circuits (relays).

  • What it does: Requests new circuit with different relays
  • What it doesn't do: Does not clear cookies, browser state, or session data
  • When to use: If website blocks based on circuit IP or relays problematic
  • When not to use: Not for regularly "rotating" identity - this creates correlation opportunities
Critical: New Identity Does Not Clear Cookies
"New Identity" only changes Tor relays. To clear cookies, close all Tor Browser windows. New Identity used incorrectly creates correlation pattern visible across multiple circuits. Use New Identity only when necessary to overcome blocking issues.

Tails (The Amnesic Incognito Live System)

Tails is privacy-focused live operating system that runs from USB stick or DVD. Tails routes all traffic through Tor, uses encrypted persistent storage (optional), and leaves no traces on the host system.

Tails Architecture

  • Live OS: Runs entirely from USB or DVD, no installation required
  • Amnesic: All data in RAM; when Tails shuts down, RAM data lost (unless persisted to encrypted drive)
  • Tor routing: All internet traffic routed through Tor by default (applications use Tor system proxy)
  • Live USB with persistence: Optional encrypted persistent storage for bookmarks, passwords, OpenPGP keys
  • Host system isolation: Tails does not access host hard drives by default

Tails Features

Default Applications (Anonymized)

  • Tor Browser: Web browsing
  • Thunderbird with TorBirdy: Email via Tor (use with encrypted email providers like ProtonMail)
  • Pidgin with OTR: Instant messaging with OTR encryption
  • KeePassXC: Password manager
  • Electrum Bitcoin Wallet: Bitcoin wallet with Tor routing
  • VLC Media Player: Media playback
  • OnionShare: File sharing via Tor
  • Liferea: RSS/Atom feed reader

Security Features

  • AppArmor profiles: SELinux/AppArmor confinement restricts application privileges
  • No swap to disk: Prevents data from being written to disk except encrypted persistence
  • Network isolation: Applications routed through Tor by default; no direct network access
  • Packages from Debian: Only vetted Debian packages included
  • No persistence by default: Prevents inadvertent data leakage across sessions

Tails USB Installation

# Installation prerequisites: # 1. Download Tails ISO from official tails.boum.org site # 2. Download Balena Etcher or Rufus # 3. Have two USB sticks (one for intermediary, one for final bootable Tails) # Installation process (on Linux/macOS/Windows): # 1. Flash Tails ISO to USB stick using Etcher # 2. Reboot to encrypted persistent storage

Tails Persistence

Tails optional persistent storage allows saving data across sessions.

  • Encrypted: Uses LUKS encryption (AES-256-XTS)
  • Password protected: Requires passphrase to access persistence
  • Optional features: Can store browser bookmarks, passwords, OpenPGP keys, SSH keys, network configurations
  • Security trade-off: Persistence creates data that could compromise anonymity if passphrase compromised or storage seized
Critical: Encryption Weakness on Storage
Tails persistence is encrypted, but encrypted data with weak passphrase or seized with passphrase is not protected. Strong passphrase requirements (minimum 20 characters, mixed case, numbers, symbols) essential. If persistence contains identifying information, adversary with access can link multiple Tails sessions.

Tails Use Cases

  • Secure browsing: Using public computers or shared devices
  • Whistleblowing: Secure drop submissions, anonymous communication
  • Journalism: Protecting sources and communications
  • Activism: Anonymous research, communication, organizing
  • Privacy: Browsing without tracking, avoiding corporate surveillance

Qubes OS Security Model

Qubes OS implements security by compartmentalization, using virtual machines to isolate different activities. Unlike Tails (live OS focused on anonymity), Qubes is permanent OS designed for security.

Qubes Architecture

Xen Hypervisor-Based Compartmentalization
Dom0: Only runs administrative code. Never connects to network. Trusted component. AppVMs: Isolated virtual machines for different activities (work, personal, banking, web browsing, Tor). Each VM has its own: - File system - Network stack (isolated or routed through proxy) - Clipboard (shared but controlled) - Applications (Linux distributions: Fedora, Debian) TemplateVMs: Base VMs for creating AppVMs. AppVMs based on TemplateVM share the same file system but are otherwise isolated.

Default Qubes AppVMs

fedora-xx (TemplateVM): Fedora-based template for AppVMs debian-xx (TemplateVM): Debian-based template for AppVMs sys-firewall: Firewall VM; handles routing for AppVMs sys-net: Network VM; handles all network traffic to/from AppVMs vault: AppVM for sensitive data (passwords, keys) work: AppVM for work personal: AppVM for personal activities untrusted: AppVM for untrusted downloads, scanning whonix-ws (Whonix Gateway): Tor proxy VM whonix-gw (Whonix Workstation): Workstation behind Whonix Gateway

Qubes Security Guarantees

  • Hypervisor isolation: Xen hypervisor enforces isolation; vulnerable application in one VM cannot escape to DOM0
  • Network isolation: Compromised VM cannot directly access network; routed through firewall VM
  • File system isolation: VMs cannot directly access each other's file systems
  • Template updates: Update templates, then restart AppVMs for automatic updates
  • Snapshots: Backup VM states and restore if compromised
Qubes vs Tails Comparison
Tails: Live OS, amnesic, simple to use, no installation, anonymity-focused, single-user Qubes: Permanent OS, requires installation, complex to learn, security-by-compartmentalization, multi-user possible (multiple identities as separate VMs) Use both: Tails for anonymity and privacy, Qubes for security and compartmentalization. They serve different purposes.

Cryptocurrency and Privacy

Bitcoin and most cryptocurrencies have public transaction ledgers visible to anyone, but privacy coins offer enhanced transaction privacy.

Bitcoin Privacy Limitations

  • Public ledger: All Bitcoin transactions recorded on blockchain visible to anyone
  • Pseudonymous: Addresses not directly linked to identity but can be correlated
  • Address reuse: Reusing addresses links multiple transactions to same user
  • Exchanges: KYC verification on exchanges links identity to addresses
  • Graph analysis: Sophisticated analysis can link transactions across addresses

Privacy Coins

Monero (XMR)

Monero is most widely adopted privacy-focused cryptocurrency:

  • Ring signatures: Transaction signers obfuscated among group of decoy signers
  • Stealth addresses: Recipient's wallet generates unique one-time addresses for each transaction
  • Confidential transactions: Transaction amounts hidden (RingCT)
  • Fungibility: All Monero coins indistinguishable from each other
  • Analysis resistance: Significantly harder to trace than Bitcoin, though not completely immune to advanced analysis

Other Privacy Coins

  • Zcash (ZEC): Zero-knowledge proofs (zk-SNARKs) hide sender, recipient, and amounts
  • Zcoin (XZC): Uses zero-knowledge proofs (Sigma) for transaction privacy
  • Dash: PrivateSend (CoinJoin mixing) option for optional privacy
  • MimbleWimble-based: Grin, Beam protocol using confidential transactions
Important: Cryptocurrency Not True Anonymous
Even privacy coins are not perfectly anonymous. Sophisticated analysis, chain analysis companies, and surveillance may reveal transactions. KYC at exchanges, poor operational security, chain analysis linking past exposures can compromise privacy. Treat anonymity as "more private" not "completely anonymous."

Secure Email and Communication

Secure email protects message content from surveillance. End-to-end encryption ensures only sender and recipient can read messages, not email providers or network eavesdroppers.

PGP/GPG Email Encryption

OpenPGP (Pretty Good Privacy/GNU Privacy Guard) provides end-to-end email encryption and digital signatures.

# Generate GPG key (Ed25519 recommended) gpg --full-generate-key # Key type: (1) RSA and RSA (legacy) # Key type: (9) ECC and ECC (recommended modern choice) # List public keys gpg --list-keys # List private keys gpg --list-secret-keys # Export public key (distribute to contacts) gpg --armor --export alice@example.com > alice_pubkey.gpg # Import public key (from others) gpg --import bob_pubkey.gpg # Encrypt file or message gpg --encrypt --armor -r bob@example.com document.txt # Decrypt gpg --decrypt document.txt.gpg > document.txt # Sign message (authentication, non-repudiation) gpg --armor --clearsign message.txt # Verify signed message gpg --verify message.txt.asc

Key management:

  • Public key: Send public key to contacts securely (verify fingerprint in person if possible)
  • Private key: Never share private key. Protect with strong passphrase.
  • Key revocation: Generate revocation certificate before use (in case key compromised)
  • Key expiration: Set expiration date; keys can be renewed

Privacy-Focused Email Providers

  • ProtonMail: End-to-end encrypted email (Switzerland-based, automatic encryption to other ProtonMail users, optional PGP for non-ProtonMail users)
  • Tutanota: End-to-end encrypted email (Germany-based, all data encrypted, automatic encryption to other Tutanota users)
  • Mailfence: End-to-end encrypted email (Belgium-based, supports PGP)
  • Disroot: Community-run federated email service (supports encryption)
Email Encryption Limitations
Email encryption protects message content but not metadata (who emailed whom, when, email subject). Encrypted emails still generate metadata. For metadata anonymity, use messaging systems designed for both content and metadata anonymity (Signal) rather than email.

Signal

  • End-to-end encrypted: Signal protocol double-ratchet algorithm provides forward secrecy and message authenticity
  • Metadata privacy: Server does not have access to message content or contact list (though some metadata visible to Signal: who is a user, when messages sent)
  • Group messages: End-to-end encrypted group chats
  • Voice and video calls: Encrypted calls
  • Verified safety numbers: Verify contact identity by comparing safety numbers (out-of-band verification recommended)
Signal vs Email Encryption
Signal: Modern, easy to use, metadata privacy (partial), end-to-end encrypted, mobile-first, no email address required PgP email: Legacy, complex to use, no metadata privacy (who-communicated-with-whom visible), requires key management, requires email provider Recommendation: Use Signal for real-time communication. Use PGP email only when Signal unavailable or email is required.

Counter-Surveillance Techniques

Counter-surveillance techniques reduce visibility to surveillance and tracking. Physical, digital, and behavioral countermeasures complement anonymity tools.

Physical Counter-Surveillance

Personal Device Security

  • Mobile phone: Smartphone constantly transmits location (cell tower triangulation, GPS, WiFi scanning)
  • Minimize tracking: Airplane mode when not in use, use VPN, limit app permissions
  • FARADAY Bags: RF-shielding bags block wireless signals (phone in bag cannot transmit or receive)
  • Burner phones: Prepaid phones purchased with cash, linked only to anonymous identity (SIM card registration may require ID in some jurisdictions)

Facial Recognition Countermeasures

  • Privacy visors: Infrared LEDs in glasses disrupt facial recognition systems without blocking normal vision
  • Face paint or masks: Disrupts facial structure recognition
  • Hairstyle and makeup: Can reduce accuracy in some systems (though increasingly less effective as systems improve)

Digital Counter-Surveillance

  • Digital hygiene: Clear cookies, cache, and browsing data regularly
  • Browser isolation: Separate anonymous browser from regular browser (profiles, virtual machines)
  • DNS over HTTPS: Use DoH to prevent DNS queries from being observed
  • DNS over TLS: Similar to DoH using TLS for DNS queries
  • HTTPS everywhere: Use browser extension to enforce HTTPS
  • Tracker blocking: uBlock Origin blocks web trackers

Behavioral Countermeasures

  • Pattern disruption: Vary times, locations, and methods to prevent pattern analysis
  • Compartmentalization: Separate identities by time (use anonymizing tools at times, personal accounts at other times)
  • Identity separation: Separate devices, accounts, and identities for different activities
  • Operational security: Consistent personas, no cross-identity references

Advanced Operational Security

Advanced OPSEC reduces correlatability and maintains separation between identities and activities.

Correlation Attack Mitigation

Sophisticated adversaries correlate identity across multiple vectors. Mitigation requires minimizing correlation opportunities:

  • Time separation: Don't switch between identities too quickly (creates time correlation)
  • Location separation: Use different VPN nodes or physical locations for different identities
  • Pattern separation: Vary browsing patterns, typing style, interests across identities
  • Device separation: Separate devices for separate identities if possible
  • Crypto wallet separation: Separate wallets for separate identities (avoid linking transactions)

Persona Consistency

Maintain consistent personas to reduce suspicion and correlation:

  • Develop backstory: Create complete persona history (job, background, interests)
  • Consistent communication: Maintain communication style, vocabulary, interests over time
  • Document persona: Record persona details to avoid inconsistencies across communications
  • Evolve naturally: Let persona evolve over time naturally (people grow and change)

Operational Security Failures

Common Advanced OPSEC Failures
Cross-site correlation: Logging into anonymous identity from same IP or device as personal identity Real-name references: Referring to real name, location, or personal details from anonymous identity Consistent language errors: Making same typing or spelling errors across all identities (linguistic fingerprinting) Payment correlation: Using same payment method (credit card, PayPal) for different identities Social media cross-referencing: Same contacts or photos across different accounts Time-of-day activity: Consistent activity at same times with same timezone (suggests physical location) Technical fingerprint: Same browser fingerprint, screen resolution, or configuration across identities
Advanced Adversary Capabilities
Nation-state adversaries with global surveillance infrastructure may correlate: - Traffic analysis across multiple networks - Correlate encrypted traffic patterns with timing and volume - Use global metadata collection to link identities - Employ sophisticated correlation algorithms and AI - Have access to browser fingerprinting databases - Perform network-level correlation across different providers Assume determined adversaries may eventually correlate identities regardless of techniques used.

Limitations and Risks

No anonymity solution provides perfect protection. Understanding limitations is crucial for realistic threat modeling.

Zero-Day Vulnerabilities

  • Browser exploits: Sophisticated adversaries may use zero-day browser exploits to deanonymize Tor users
  • Tor vulnerabilities: Potential for Tor relay compromise or protocol vulnerabilities
  • De-anonymization attacks: Traffic analysis, correlation attacks, browser fingerprinting

User Error

  • Identity leakage: Accidentally revealing real identity (personal references, shared passwords)
  • Configuration errors: Misconfiguring anonymity tools (e.g., logging into personal accounts while using anonymizing tools)
  • Logging: Using Tor for anonymity but using other software that logs (e.g., using same web browser for both)

Legal Risks

  • Anonymous speech laws: Some jurisdictions restrict anonymous speech or require identification
  • Anonymous online behavior may still be illegal: Anonymity doesn't legalize illegal activities
  • Tor exit node monitoring: Law enforcement monitors exit node traffic for illegal content

Risk Assessment

Apply anonymity tools based on threat model:

  • Low threat (privacy from corporation tracking): Tor Browser, privacy-focused browser ( Brave, Firefox with anti-tracking), VPN
  • Medium threat (government surveillance): Tor, Tails, compartmentalized identities, privacy-focused services
  • High threat (nation-state targeting): Tails, Qubes, multiple anonymity layers, operational security, compartmentalization, assume eventual compromise
Critical: No Perfect Anonymity
No anonymity tool or technique provides absolute protection against all adversaries, particularly nation-states. Assume sophisticated adversaries with sufficient resources and global access may eventually correlate identities. Anonymity increases difficulty and cost, not impossibility of identification. Responsible use requires legal compliance and ethical considerations.
