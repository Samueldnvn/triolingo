MODULE 10

Defensive Strategies

Introduction

Defensive strategies integrate all cybersecurity capabilities into comprehensive, layered protection. Defense in depth, security frameworks, continuous monitoring, and continuous improvement form the foundation of organizational resilience. This final module synthesizes concepts from all previous modules into practical, implementable defensive strategies.

Critical: Defense in Depth

Security Frameworks

Security frameworks provide structured approaches to implementing and managing cybersecurity programs. Frameworks translate abstract security goals into actionable controls and metrics.

NIST Cybersecurity Framework

NIST CSF Functions
Identify: Develop understanding to manage cybersecurity risk (asset management, business environment, governance, risk assessment) Protect: Develop and implement safeguards (identity management, access control, data security, protective technology) Detect: Identify cybersecurity events (anomalies and events, security continuous monitoring, detection processes) Respond: Take action (response planning, communications, analysis, mitigation, improvements) Recover: Restore capabilities (recovery planning, improvements, communications

ISO/IEC 27001

ISO/IEC 27001 is international standard for information security management systems (ISMS). Certification demonstrates compliance with international best practices.

  • ISMS: Information Security Management System - systematic approach to managing sensitive information
  • Controls: 114 controls across 14 domains (information security policies, organization of information security, human resource security)
  • Certification: External audit and certification process
  • Benefits: International recognition, customer confidence, regulatory compliance
  • Scope: Applicable to any organization requiring formalized security program

CIS Controls

Center for Internet Security (CIS) Controls provide prioritized list of security actions.

CIS v8 Controls (Prioritized)

  • Critical Controls (IG1): "Cyber Hygiene" - foundational for all organizations
  • Advanced Controls (IG2): Additional controls for organizations with higher security requirements
  • Expert Controls (IG3): Additional controls for organizations with mature security programs
CIS v8 Top Controls
01: Inventory and Control of Enterprise Assets 02: Inventory and Control of Software Assets 03: Data Protection 04: Secure Configuration 05: Account Management 06: Access Control Management 07: Continuous Vulnerability Management 08: Audit Log Management 09: Email and Web Browser Protections 10: Malware Defenses 11: Data Recovery 12: Network Infrastructure Management 13: Network Device Monitoring 14: Network Segmentation 15: Wireless Access Control 16: Application Security 17: Incident Response Management 18: Penetration Testing

Choosing a Framework

  • Mandatory frameworks: Some industries require specific frameworks (PCI DSS for payment cards, HIPAA for healthcare, SOX for publicly traded companies)
  • Voluntary frameworks: NIST CSF, ISO 27001, CIS Controls - choose based on organizational maturity, risk profile, and customer requirements
  • Hybrid approaches: Many organizations use multiple frameworks (NIST CSF for internal program, ISO 27001 for customer demonstrations)
  • Size considerations: Smaller organizations may use CIS Critical Controls; larger organizations may implement ISO 27001 certification

Security Monitoring and SOC

Security monitoring provides visibility into security posture, detects incidents early, and enables timely response. Security Operations Center (SOC) is central hub for monitoring, detection, and response.

SOC Functions

Level 1 (Tier 1): SOC Analysts

  • Monitor security tools and SIEM alerts
  • Triage alerts (investigate, classify, escalate if necessary)
  • Handle simple security incidents (password resets, account lockouts)
  • Escalate complex or critical incidents to Level 2

Level 2 (Tier 2): Security Analysts

  • Investigate escalated incidents from Level 1
  • Perform forensic analysis and evidence collection
  • Perform threat hunting (proactively search for threats)
  • Lead containment and remediation for incidents

Level 3 (Tier 3): Senior Security Analysts

  • Handle most complex security incidents
  • Perform advanced forensics and malware analysis
  • Develop detection rules and playbooks
  • Liaise with law enforcement and external vendors

SOC Manager

  • Overall SOC operations and strategy
  • Resource allocation and staffing
  • KPI tracking and process improvement
  • Liaison with other departments (IT, legal, PR)

SOC Metrics and KPIs

  • Mean Time to Detect (MTTD): Average time from incident start to detection
  • Mean Time to Respond (MTTR): Average time to contain and remediate incidents
  • False positive rate: Percentage of alerts that are not actual incidents (target: minimize)
  • Alert triage time: Average time to investigate and classify alerts
  • Incident closure rate: Percentage of incidents resolved within SLA
SOC Maturity Levels
Initial: Ad-hoc monitoring, no formalized processes Repeatable: Formalized processes and procedures exist Defined: Comprehensive security program, documented procedures Managed: Security controls measured, quantitative improvement tracked Optimized: Continuous improvement, optimized processes, automation

Managed SOC Options

  • In-house SOC: Organization builds internal SOC (highest control, highest cost)
  • Managed Security Service Provider (MSSP): Outsource monitoring to provider (lower cost, less control)
  • Hybrid: In-house SOC for critical response, MSSP for routine monitoring
  • MSP vs MSSP: MSP provides IT services; MSSP specifically provides security monitoring

Threat Intelligence

Threat intelligence provides context about potential and current threats, enabling proactive defense. Effective threat intelligence transforms raw data into actionable intelligence.

Threat Intelligence Levels

Pyramid of Pain (Intelligence Types)
Hash values: File and binary hashes. Easiest for adversaries to change, provides least long-term value. IP addresses: C2 server IPs, scan sources. Moderately easy for adversary to change. Domain names: C2 domains, phishing domains. Harder for adversary to change. Network/host artifacts: Registry keys, file paths, mutexes. Difficult for adversary to change. Tools: Malware toolkits used. Harder for adversary to change. TTPs (Tactics, Techniques, Procedures): Attack methodology. Most difficult for adversary to change - provides most long-term detection value.

Threat Intelligence Sources

  • Open Source Intelligence (OSINT): Vulnerability feeds (CVE), CISA alerts, security blogs, social media (Twitter #infosec)
  • Commercial threat intelligence: Recorded Future, Anomali ThreatStream, CrowdStrike Falcon X
  • Information sharing: ISACs (Information Sharing and Analysis Centers), FS-ISAC, Auto-ISAC
  • Government sources: CISA, FBI InfraGard, NCSC (UK), CERTs
  • In-house threat intelligence: Internal telemetry, incident history

MITRE ATT&CK Framework

MITRE ATT&CK is adversary tactics and techniques knowledge base used for threat hunting and detection rule development.

  • Adversary emulation: Use ATT&CK tactics to test detection capabilities
  • Detection rule development: Map ATT&CK techniques to SIEM and EDR detection rules
  • Threat hunting: Search for evidence of specific ATT&CK techniques
  • Red teaming: Red teams select ATT&CK techniques to emulate realistic adversary behaviors

Vulnerability Management

Vulnerability management identifies, classifies, prioritizes, and remediates vulnerabilities to reduce attack surface. Effective vulnerability management requires continuous assessment and remediation processes.

Vulnerability Detection

Vulnerability Scanning

  • Credentials required: Scan requires credentials to log into hosts for accurate vulnerability detection
  • Port scanning: Identifies open ports and services
  • Vulnerability identification: Matches services against vulnerability database
  • Tools: Nessus, Qualys, Rapid7, OpenVAS

Penetration Testing

  • Authorized, ethical hacking: Simulates real-world attack to identify exploitable vulnerabilities
  • Types: Black box (no prior knowledge), Grey box (partial knowledge), White box (full knowledge)
  • Internal vs external: External tests internet-facing infrastructure; internal tests internal network access
  • Value: Finds logic errors and vulnerabilities scanners miss
  • Reporting: Comprehensive report with findings, risk ratings, and remediation recommendations

Vulnerability Prioritization

CVSS v3 Scoring
CVSS (Common Vulnerability Scoring System) standardizes vulnerability severity ratings Base score (0.0 - 10.0): Intrinsic vulnerability characteristics (exploitability impact) Severity levels: - None: 0.0 - Low: 0.1 - 3.9 - Medium: 4.0 - 6.9 - High: 7.0 - 8.9 - Critical: 9.0 - 10.0
  • Prioritization factors: CVSS score, exploit availability, criticality of affected system, ease of exploitation, potential impact
  • Risk-based approach: Prioritize vulnerabilities that matter most to your organization (not just highest CVSS)
  • Zero-day vulnerabilities: Immediate priority if exploit available (no patch exists)

Remediation

  • Software updates: Patch to vulnerable version (remediation)
  • Workarounds: Disable vulnerable service (mitigation) until patch available
  • Compensating controls: Add network segmentation, additional monitoring (risk acceptance)
  • Exceptions: Documented exceptions for critical systems that cannot be patched immediately
Critical: Patch Management
Effective vulnerability management requires prioritization and risk-based approach. Not all vulnerabilities need immediate remediation. Apply patches based on CVSS score, exploit availability, criticality of affected system, and organizational risk tolerance. Document exceptions and acceptance of risk when system cannot be patched.

Security Awareness Training

Security awareness program trains employees to recognize and respond to security threats. People are often the weakest link in security - effective training reduces employee vulnerabilities.

Training Components

Initial Training

  • Security policies and procedures
  • Phishing recognition (suspicious emails, URLs, attachments)
  • Data handling and classification
  • Physical security (badges, visitors, unattended devices)
  • Mobile device security (BYOD policies, lost devices)

Regular Refreshers

  • Quarterly or monthly micro-training (5-10 minutes)
  • Current threat landscape updates
  • New policies and procedures
  • Real-world incident case studies

Phishing Simulations

  • Simulated phishing attacks sent to employees
  • Measure click rates and report rates
  • Provide immediate feedback if employee clicks (educational, not punitive)
  • Track improvement over time
  • Simulate various phishing types (deceptive, spear phishing, Business Email Compromise)

Building Security Culture

  • Leadership support: Executives must model security-conscious behavior
  • No blame culture: Focus on learning, not punishment for false positives or minor failures
  • Recognition: Reward and recognize security-conscious behavior
  • Engagement: Make training relevant and engaging (gamification, real-world examples)
  • Metrics: Track and report security awareness improvements
Training Frequency
Initial: Onboarding training for all employees Periodic: Annual refresher training Ongoing: Monthly micro-training or quarterly updates Triggered: When new threats emerge, immediately communicate with employees Training alone insufficient without supportive security culture and clear policies.

Third-Party Risk Management

Third-party risk management identifies, assesses, and mitigates risk from vendors, suppliers, and partners. Third parties are increasingly common attack vectors (supply chain attacks).

Critical: Supply Chain Attacks
Supply chain attacks target software vendors, service providers, or third-party components to compromise multiple organizations simultaneously. Examples: - SolarWinds (2020): Compromise of IT management software affected thousands of organizations - Kaseya (2021): MSP compromise affected downstream customers - Log4Shell (2021): Vulnerability in widely-used Java library Assess third-party security before integration and continuously monitor vendor security.

Third-Party Risk Assessment

  • Vendor questionnaire: Assess security practices, certifications, controls
  • Security review: Review vendor security documentation (SOC 2 reports, penetration tests)
  • Risk classification: Classify vendors by risk tier based on access to data, criticality
  • Contractual requirements: Include security requirements in vendor contracts
  • Continuous monitoring: Monitor for vendor data breaches, CVEs in vendor software

Vendor Tiering

Vendor Tiers (by risk and criticality): Tier 1 (Critical): - Access to sensitive data - Access to production systems - High financial/exposure if compromised - Require annual audit, penetration testing Tier 2 (High): - Access to non-sensitive data - Access to non-critical systems - Medium exposure if compromised - Require questionnaires, periodic review Tier 3 (Moderate): - Limited access - Minimal exposure - Require basic assessment Tier 4 (Low): - No access to data or systems - Low exposure - Minimal requirements

Software Supply Chain Security

  • Supply chain attack prevention: Vet open source dependencies, sign all software artifacts
  • Software composition analysis (SCA): Automated tool to scan code for third-party dependencies and known vulnerabilities
  • CI/CD security: Secure software development pipelines (branch protection, code review, signing artifacts)
  • SBOM (Software Bill of Materials): List of all software components in delivered product (enables rapid CVE response)

Zero Trust Architecture

Zero Trust is security model based on principle "never trust, always verify." Zero Trust assumes breach and validates every request regardless of origin. Traditional perimeter-based security is replaced by per-request authentication and authorization.

Zero Trust Principles

  • Verify explicitly: Always authenticate and authorize based on all available data points (identity, location, device, service)
  • Least privilege access: Grant minimum necessary access (just-in-time, just-enough access)
  • Assume breach: Design assuming adversary is already inside network
  • Microsegmentation: Segment network to limit lateral movement
  • Continuous monitoring: Monitor and verify security posture continuously

Zero Trust Implementation

Identity Verification

  • MFA for all access: No exceptions, including internal network access
  • Conditional access policies: Require MFA for sensitive operations, block risky devices
  • Identity protection: Detect and respond to compromised credentials and accounts
  • Privileged access management (PAM): Just-in-time elevation for admin access

Device Health

  • Device compliance: Verify device health (patched, antivirus, security policies)
  • Device trust: Trust unknown/non-corporate devices less than known/corporate devices
  • BYOD policies: MDM (Mobile Device Management) for personal devices accessing corporate resources

Network Segmentation

  • Microsegmentation: Segment at application and workload level (not just network level)
  • Software-defined perimeters (SDP): Control access to applications before network connectivity established
  • Isolation: Isolate critical systems, databases, and applications

Zero Trust Benefits

  • Reduced attack surface: Compromise of one system does not automatically grant access to others
  • Limited lateral movement: Attacker movement inside network constrained by continuous verification
  • Modern security posture: Accounts for cloud, mobile access, and remote work (traditional perimeter model insufficient)
  • Reduced risk from credential theft: MFA and device verification reduce impact of password compromise
Adoption Consideration
Zero Trust is modern security model addressing current threat landscape and computing environment (cloud, mobile, remote work). Traditional perimeter-based security insufficient in modern environments. Implement Zero Trust gradually, starting with highest-risk systems and expanding over time.

Measuring Security Success

Security effectiveness requires measurement. Metrics enable improvement, communicate security posture to leadership, and justify investment.

Leading vs Lagging Indicators

Indicator Types
Lagging indicators: Measure what has happened (incidents, breaches, vulnerabilities patched). - Number of security incidents - Time to patch critical vulnerabilities - Number of phishing attacks successful - Compliance audit results Leading indicators: Predict future security posture. - Security awareness training completion rates - MFA adoption rate - Vulnerability scan coverage - Patch management coverage - Number of security controls implemented

Key Security Metrics

  • MTTD and MTTR: Mean Time to Detect and Mean Time to Respond (lower is better)
  • Vulnerability remediation SLA: Percentage of critical vulnerabilities patched within SLA
  • Phishing simulation results: Click rate improvement over time
  • Training completion: Percentage of employees completing security awareness training
  • Security controls coverage: Percentage of systems with required controls in place
  • False positive rate: Percentage of alerts that are false positives (target: minimize)

Risk-Based Metrics

  • Risk reduction over time: Quantified reduction in cyber risk (qualitative or quantitative)
  • Cybersecurity insurance premiums: Lower premiums indicate improved security posture
  • Audit findings: Number of findings decreasing over time
  • Penetration testing results: Number and severity of vulnerabilities decreasing over time

Continuous Improvement

Security is continuous improvement process, not one-time goal. Threats evolve, new vulnerabilities emerge, and technology changes. Mature security organizations continuously adapt and improve.

Plan-Do-Check-Act (PDCA) Cycle

Plan

  • Assess current security posture
  • Establish security objectives and targets
  • Create plans to address gaps
  • Align with business objectives

Do

  • Implement security controls and policies
  • Train employees
  • Deploy technology solutions

Check

  • Measure performance against objectives
  • Conduct audits and reviews
  • Review incident metrics
  • Assess emerging threats and vulnerabilities

Act

  • Take corrective actions on deficiencies
  • Adjust plans as necessary
  • Update policies and procedures
  • Cycle back to Plan phase

Feedback Loops

  • Incident lessons learned: Incorporate incident learnings into future prevention
  • Threat intelligence feedback: Use threat intelligence to prioritize improvements
  • Employee feedback: Solicit feedback from employees on security policies and tools
  • Leadership feedback: Align security investments with business risk priorities

Security Roadmap

Create multi-year security roadmap for improvement:

  • Current state assessment: Gap analysis against frameworks (NIST CSF, ISO 27001, CIS Controls)
  • Future state: Target maturity level and capability goals
  • Prioritized initiatives: Order initiatives by risk and business impact
  • Milestones: Timeline and milestones for achieving goals
  • Resource allocation: Budget and personnel requirements
