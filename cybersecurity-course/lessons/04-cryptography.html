MODULE 4

Cryptography

Introduction

Cryptography is the science of protecting information through encryption, enabling secure communication, data integrity, authentication, and non-repudiation. Understanding modern cryptographic primitives, protocols, and implementation pitfalls is essential for building and auditing secure systems. This module covers symmetric encryption, public key cryptography, hash functions, digital signatures, and the infrastructure that ties them together.

Critical Warning
Do not implement your own cryptography. Use well-vetted, widely-analyzed cryptographic libraries. Even experts make mistakes implementing cryptography. The primitives themselves are correct; the weaknesses almost always lie in implementation details, key management, and protocol design.

Cryptographic Fundamentals

Cryptography provides security services at different layers of the security model. Understanding these services and when to apply them is fundamental to secure system design.

Cryptographic Goals

  • Confidentiality: Preventing unauthorized parties from reading data (achieved through encryption)
  • Integrity: Ensuring data has not been modified (achieved through hash functions or message authentication codes)
  • Authentication: Verifying identity of communicating parties (achieved through digital signatures, MACs, or authentication protocols)
  • Non-repudiation: Proving that a party took an action (achieved through digital signatures with public keys)

Terminology

Key Terms
Plaintext: Unencrypted, readable data Ciphertext: Encrypted, unreadable data Encryption: Process of transforming plaintext to ciphertext Decryption: Process of transforming ciphertext back to plaintext Key: Secret value used with algorithm for encryption/decryption Key space: Set of all possible keys (larger key space = stronger against brute force) Salt: Random data added to password before hashing to prevent rainbow table attacks

Kerckhoff's Principle

Kerckhoff's principle states that security should reside in the secrecy of the key, not the secrecy of the algorithm. Cryptographic algorithms should be public and well-analyzed.

Why Public Algorithms?
- Open analysis: More experts can analyze and find weaknesses - Trust through scrutiny: Algorithms withstand years of global analysis gain trust - No security through obscurity: Secret algorithms can be reverse-engineered - Implementation transparency: Open standards allow verified implementations Examples: AES was selected through public competition. RSA security relies on mathematical hardness, not secrecy.

Shannon's Theorem - Perfect Secrecy

Claude Shannon proved that perfect secrecy requires a key as long as the message. In practice, this is impractical for most applications. Modern cryptography trades perfect secrecy for practical, "computationally secure" systems.

Computational security: An algorithm is secure if breaking it would require computational resources beyond current or foreseeable capabilities (e.g., requiring millions of years with current computers).

Symmetric Encryption

Symmetric encryption uses the same key for both encryption and decryption. It is faster than asymmetric encryption but requires secure key exchange between parties. symmetric encryption is ideal for encrypting large volumes of data.

Symmetric Encryption Process
Alice wants to send encrypted message to Bob: 1. Alice and Bob share secret key (via secure channel or key exchange protocol) 2. Alice encrypts: `ciphertext = encrypt(key, plaintext)` 3. Alice sends ciphertext to Bob (can be over insecure channel) 4. Bob decrypts: `plaintext = decrypt(key, ciphertext)` Eve (adversary) sees ciphertext but cannot decrypt without key

AES (Advanced Encryption Standard)

AES is the most widely used symmetric encryption algorithm. Selected through NIST competition in 2001, AES supports 128-bit, 192-bit, and 256-bit keys.

AES Architecture

AES encrypts data in 128-bit blocks using substitution-permutation network:

  • SubBytes: Byte substitution using S-box (non-linear transformation)
  • ShiftRows: Row shifting (linear transformation)
  • MixColumns: Column mixing (linear transformation)
  • AddRoundKey: XOR with round key

Number of rounds depends on key size:

Key Size Rounds Security Level
AES-128 10 Suitable for most applications
AES-192 12 Higher security margin
AES-256 14 Maximum recommended future-proofing

AES Modes of Operation

Block cipher modes define how AES processes messages longer than one block (128 bits):

ECB (Electronic Codebook)

Encrypts each block independently with same key:

  • Pros: Simple, parallelizable
  • Cons: Identical plaintext blocks produce identical ciphertext (pattern leakage)
  • Vulnerable: Not semantically secure, ECB reveals patterns
Critical: Do Not Use ECB
ECB is never secure for real data. Images encrypted with ECB still reveal visual patterns. ECB fails the definition of encryption because adversaries can determine when two messages start with same prefix. Never use ECB.

CBC (Cipher Block Chaining)

Each plaintext block is XORed with previous ciphertext block before encryption:

C1 = Encrypt(K, P1 XOR IV) C2 = Encrypt(K, P2 XOR C1) C3 = Encrypt(K, P3 XOR C2) ...
  • IV (Initialization Vector): Random value for first block, must be unpredictable
  • Pros: Same plaintext encrypts to different ciphertext each time
  • Cons: Not parallelizable (encryption only), IV reuse breaks security
  • Vulnerable: Bit-flipping attacks can modify ciphertext in predictable ways

CTR (Counter Mode)

Encrypts a counter value to produce keystream, XORed with plaintext:

Keystream = Encrypt(K, Counter) Ciphertext = Plaintext XOR Keystream
  • Nonce: Counter starting value (must never repeat with same key)
  • Pros: Parallelizable (both encryption and decryption), turns block cipher into stream cipher
  • Cons: Counter reuse catastrophic (XORing keystream with different messages reveals both plaintexts)

GCM (Galois/Counter Mode)

Combines CTR encryption with MAC for authentication:

  • Authenticated Encryption: Provides both confidentiality and integrity (AEAD)
  • Pros: Efficient, widely recommended, parallelizable
  • Cons: None significant for most applications
  • Default choice: AES-GCM is the default mode for new systems using authenticated encryption
Recommendation
For new development: - Default: AES-256-GCM for authenticated encryption - Alternative: ChaCha20-Poly1305 if AES not available or performance critical - Avoid: ECB, CBC without MAC (CBC-then-MAC or MAC-then-Encrypt is error-prone) Authenticated encryption modes (GCM, Poly1305) handle both confidentiality and integrity correctly, avoiding implementation pitfalls.

ChaCha20-Poly1305

ChaCha20-Poly1305 is a modern alternative when AES acceleration is unavailable. Widely used in TLS (by default in many clients) and mobile applications.

  • ChaCha20: Stream cipher providing confidentiality
  • Poly1305: Message authentication code providing integrity
  • Advantages: Fast in software, no special hardware required, simpler implementation
  • Security: Proven security reduction, widely analyzed

Asymmetric (Public Key) Encryption

Asymmetric encryption uses different keys for encryption and decryption: public key (shared openly) and private key (kept secret). This enables secure communication without pre-shared keys and enables digital signatures.

Asymmetric Encryption Process
Alice wants to send encrypted message to Bob: 1. Bob publishes public key (accessible to anyone) 2. Alice encrypts using Bob's public key: `ciphertext = encrypt(Bob_Public_Key, plaintext)` 3. Alice sends ciphertext to Bob 4. Bob decrypts using private key: `plaintext = decrypt(Bob_Private_Key, ciphertext)` Eve (adversary) has public key but cannot decrypt without private key

RSA

RSA, invented by Rivest, Shamir, and Adleman in 1977, is the most well-known public-key cryptosystem. RSA security relies on the difficulty of factoring large integers.

RSA Key Generation

  1. Choose two large primes p and q (typically 1024+ bits each)
  2. Compute n = p × q (this is the modulus)
  3. Compute φ(n) = (p-1) × (q-1)
  4. Choose public exponent e (typically 65537) where 1 < e < φ(n) and gcd(e, φ(n)) = 1
  5. Compute private exponent d where e × d ≡ 1 mod φ(n)
  6. Public key: (n, e), Private key: (n, d)

RSA Encryption and Decryption

Encryption: ciphertext = plaintext^e mod n

Decryption: plaintext = ciphertext^d mod n

RSA Key Sizes

Key Size Security Level Recommendation
1024 bits ~80 bits (obsolete) Not recommended - factorable with moderate resources
2048 bits ~112 bits Minimum recommended (adequate through 2030)
3072 bits ~128 bits Recommended for higher security requirements
4096 bits ~150 bits Maximum recommended size beyond which performance costs increase
RSA Limitations
RSA is slow. 2048-bit RSA keys are orders of magnitude slower than AES-256 encryption. Practically, RSA is used: - To encrypt symmetric keys (hybrid encryption), not large data - For digital signatures (signing hashes, not full messages) - For key exchange protocols

Elliptic Curve Cryptography (ECC)

Elliptic curve cryptography provides comparable security to RSA with much smaller keys, offering better performance and bandwidth efficiency. ECC security relies on the elliptic curve discrete logarithm problem (ECDLP).

Key Size Comparison

Security Level RSA Key Size ECC Key Size
80 bits 1024 160-224
112 bits 2048 224-255
128 bits 3072 256-383
256 bits 15360 512+

Common Elliptic Curves

  • P-256 (secp256r1): NIST curve, broadly supported, ~128-bit security
  • P-384 (secp384r1): NIST curve, higher security ~192 bits
  • P-521 (secp521r1): NIST curve, ~254-bit security
  • Curve25519: Modern curve, ~128 bits, fast, designed for speed
  • ed25519: Edwards curve for digital signatures
Recommendation
For new development, prefer ECC over RSA: - Default: P-256 for compatibility, P-384 for higher security - Modern choice: Ed25519 for signatures, X25519 (Curve25519) for key exchange - Benefits: Smaller keys, faster computation, lower bandwidth Curve25519 and ed25519 are widely recommended for new applications due to careful design and simplicity.

Key Exchange

Key exchange protocols allow two parties to securely establish shared secret over insecure channel. Diffie-Hellman is the foundational key exchange protocol.

Diffie-Hellman (DH)

DH allows Alice and Bob to derive shared secret without transmitting it:

Public parameters: G (generator), P (large prime) Alice: Choose private key a (random) Compute public key A = G^a mod P Send A to Bob Bob: Choose private key b (random) Compute public key B = G^b mod P Send B to Alice Shared secret: Alice computes: S = B^a mod P = G^(ba) mod P Bob computes: S = A^b mod P = G^(ab) mod P Alice and Bob both compute S = G^(ab) mod P (same value)
  • Vulnerable: Man-in-the-middle attacks unless combined with authentication
  • Used: In TLS (Diffie-Hellman key exchange)
  • Modern: ECDH (Elliptic Curve Diffie-Hellman) is preferred for better performance

Forward Secrecy

Forward secrecy (or perfect forward secrecy) ensures that compromise of long-term private keys does not compromise past session keys:

  • Traditional RSA key exchange: If RSA private key compromised, all past communications recorded by adversary can be decrypted
  • Diffie-Hellman with ephemeral keys (DHE/ECDHE): Session keys derived independently each session, long-term keys only used for authentication, not key derivation
  • Protection: Compromised long-term keys cannot decrypt past sessions
Recommendation
Always use ephemeral key exchange (ECDHE) when possible. TLS should configure: - ECDHE key exchange (ephemeral) - RSA or ECDSA for server authentication This combination provides forward secrecy: compromise of server private key does not allow decryption of past TLS sessions recorded by adversary.

Hash Functions

Cryptographic hash functions produce fixed-size output (digest) from arbitrary-size input. Hash functions provide integrity verification, password storage, and are building blocks for MACs, digital signatures, and random number generation.

Properties of Cryptographic Hash Functions

  • Preimage resistance: Given hash output H, infeasible to find input M where hash(M) = H
  • Second preimage resistance: Given input M1, infeasible to find different input M2 where hash(M1) = hash(M2)
  • Collision resistance: Infeasible to find any two inputs M1 and M2 where hash(M1) = hash(M2)

Common Hash Functions

Algorithm Output Size Status
MD5 128 bits BROKEN - Collisions trivially computable
SHA-1 160 bits BROKEN - Collisions demonstrated (SHAttered attack)
SHA-256 256 bits Secure - Widely recommended
SHA-512 512 bits Secure - Higher security, faster on 64-bit systems
SHA-3 224/256/384/512 Secure - NIST standard, different construction than SHA-2
Critical: Avoid Broken Hashes
MD5 and SHA-1 are completely broken for security purposes. Practical collision attacks exist and can produce two files with same hash. Never use MD5 or SHA-1 for any security function. Use SHA-256 or SHA-3 (Keccak).

Password Hashing

Passwords require specialized hashing because:

  • Passwords are low-entropy (limited character set, often short)
  • Passwords are re-used frequently (credential stuffing)
  • Attackers can use fast hash functions to brute force passwords

Password hashing must be:

  • Slow: Deliberately computationally expensive to slow brute force
  • Salted: Random salt prevents rainbow table attacks (pre-computed hash tables)
  • Memory-hard: Require significant memory, increasing cost of GPU/ASIC attacks

Recommended Password Hash Algorithms

Argon2

  • Winner: Password Hashing Competition 2015
  • Types: Argon2d (data-dependent), Argon2i (data-independent), Argon2id (hybrid, recommended)
  • Configurable: Memory cost, time cost, parallelism
  • Status: NIST-recommended, widely supported
  • Default choice: Argon2id for password storage

bcrypt

  • Long-standing: Since 1999, extensively analyzed
  • Work factor: Adjustable cost parameter (2^work_factor iterations)
  • Built-in salt: Automatically handles salts
  • Limitations: 72-byte password limit (solved by pre-hashing)
  • Status: Still secure, widely used

PBKDF2

  • Standard: NIST standard (PKCS #5)
  • Iterations: Configurable iteration count
  • Algorithm: Can use HMAC with SHA-256, SHA-512, etc.
  • Status: Secure but less resistant to GPU/ASIC attacks than Argon2 or scrypt
  • Use case: When support for Argon2 or bcrypt unavailable
Never Use Fast Hashes for Passwords
Never use MD5, SHA-1, SHA-256, SHA-512, or other fast hashes to store passwords. Adversary can hash billions of passwords per second on GPUs. Always use Argon2id, bcrypt, or PBKDF2 (prefer Argon2id).

MAC (Message Authentication Code)

MACs provide integrity and authentication for messages using secret key. HMAC construction builds MAC from hash function.

HMAC(K, M) = hash((K ⊕ opad) || hash((K ⊕ ipad) || M))

Key terms:

  • Key (K): Secret key shared by sender and receiver
  • Message (M): Data being authenticated
  • hash: Hash function (SHA-256, SHA-3)
  • iPad: Inner padding (0x36 repeated)
  • oPad: Outer padding (0x5c repeated)
HMAC Usage Example
Alice sends message with MAC to Bob: 1. Alice computes: MAC = HMAC(K, M) 2. Alice sends (M, MAC) to Bob 3. Bob computes: MAC' = HMAC(K, M) 4. Bob verifies: MAC == MAC' If message M tampered during transmission, MAC does not match, Bob rejects message. Adversary without key K cannot compute valid MAC for modified message.

Public Key Infrastructure (PKI)

Public Key Infrastructure is the system that binds public keys to entities (organizations, individuals, servers) through digital certificates issued by Certificate Authorities. PKI enables encryption and authentication at scale.

X.509 Certificates

X.509 is the standard format for digital certificates used in HTTPS (TLS), code signing, email encryption (S/MIME), and many other applications.

Certificate Structure

Field Purpose
Version Certificate format version (usually v3)
Serial Number Unique identifier for certificate issued by CA
Issuer Certificate Authority that issued this certificate
Subject Entity certificate represents (e.g., example.com)
Validity Not Before and Not After dates
Public Key Subject's public key (and algorithm)
Issuer Signature Signature over previous fields, signed by CA
Extensions Additional info (Subject Alternative Name, Key Usage, etc.)

Certificate Chain and Validation

Certificate Chain
Root CA (self-signed, pre-installed in browsers/operating systems) | ├─> Intermediate CA (signed by Root CA) | └─> Server Certificate (signed by Intermediate CA) Validation process: 1. Verify server certificate signature using intermediate CA public key 2. Verify intermediate CA signature using root CA public key 3. Verify root CA signature (self-signed) against trusted root store 4. Check validity (not expired) 5. Check revocation (not in CRL or OCSP response)

Certificate Revocation

Certificates may need to be revoked before expiration (key compromise, CA compromise, change in ownership, fraudulent certificate).

  • CRL (Certificate Revocation List): List of revoked serial numbers published by CA
  • OCSP (Online Certificate Status Protocol): Real-time query to CA for revocation status
  • OCSP Stapling: Server includes OCSP response in TLS handshake

Let's Encrypt

  • Non-profit CA: Provides free X.509 certificates
  • Automated: ACME protocol for automated certificate issuance
  • 90-day validity: Short lifetime encourages automation, limits damage if compromised
  • Universal trust: Trusted by all major browsers
  • Impact: Dramatically increased HTTPS adoption by removing cost barrier
Production Deployment
For production HTTPS: - CA: Use Let's Encrypt (free, automated) or commercial CA (better support) - Renewals: Automate with certbot or similar ACME client - Monitoring: Monitor certificate expiration (set alerts 30 days before)
