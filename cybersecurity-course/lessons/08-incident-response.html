MODULE 8

Incident Response & Forensics

Introduction

Incident response is the systematic approach to preparing for, detecting, containing, and recovering from security incidents. Digital forensics involves collecting, preserving, and analyzing evidence to identify what happened, how it happened, and who was responsible. Together, incident response and forensics minimize damage while preserving evidence for legal and remediation purposes.

Critical: Preparedness is Essential
Organizations that prepare for incidents before they occur respond more effectively, minimize damage, and recover faster. Incident response cannot be improvised in the moment. Develop incident response plans, train teams, and test procedures regularly. The time to prepare is before the incident, not during.

Incident Response Lifecycle

NIST Special Publication 800-61 defines the incident response lifecycle with four phases. Understanding these phases provides framework for systematic incident handling.

NIST Incident Response Phases
1. Preparation: Establish and train incident response team, develop procedures, acquire tools 2. Detection and Analysis: Detect incidents, triage, identify scope and impact, determine containment strategy 3. Containment, Eradication, and Recovery: Contain incident to prevent spread, eradicate attacker presence, restore normal operations 4. Post-Incident Activity: Lessons learned, document incident, update procedures, improve defenses

Phase 1: Preparation

Preparation occurs before incidents happen. Good preparation dramatically improves response effectiveness.

Incident Response Team

  • Incident Response Manager (IRM): Overall coordination, decision-making, communications
  • Technical leads: Forensic analysts, network security, security operations center analysts
  • Legal counsel: Legal implications, regulatory reporting, law enforcement coordination
  • Public relations/Communications: Public messaging, stakeholder communications
  • Human resources: Employee communications, disciplinary actions, personnel involved
  • IT operations: System recovery, restoration, technical support

Incident Response Plan

  • Incident classification: Define incident categories and severity levels
  • Contact information: Team members, vendors, law enforcement, regulators contact details
  • Communication procedures: How to notify team, escalation procedures
  • Evidence collection: Procedures for preserving evidence, chain of custody
  • Containment procedures: Pre-approved containment strategies for different incident types
  • Recovery procedures: System restoration from backups, network reconfiguration

Tools and Services

  • SIEM: Security Information and Event Management - centralized log analysis
  • EDR: Endpoint Detection and Response - endpoint monitoring and response
  • Network monitoring: IDS/IPS, packet capture, netflow analysis
  • Forensics tools: Disk imaging, memory forensics, timeline analysis
  • Retained services: Forensic services provider on retainer for major incidents

Phase 2: Detection and Analysis

Prompt detection limits damage. Analysis determines containment and recovery strategy.

Detection Sources

  • Automated alerts: IDS/IPS, SIEM rules, EDR alerts
  • User reports: Employees reporting suspicious emails, unusual activity, security concerns
  • External notifications: Third-party notifications (e.g., "your servers participating in DDoS")
  • Security monitoring: Log analysis, anomaly detection, threat intelligence feeds
  • System behavior: Unexpected reboots, performance issues, unusual network traffic

Incident Triage

Initial assessment to determine severity and required response:

  • Incident confirmed? Is this real or false positive?
  • Type of incident: Malware, data breach, denial-of-service, unauthorized access, social engineering?
  • Scope affected: Which systems, networks, data?
  • Impact: Data exposure, financial loss, operational impact, reputation?
  • Timeframe: When did incident start? How long has attacker had access?

Evidence Collection (During Analysis)

# Linux system information collection # System time (for timeline) date # Logged-in users who -u w # Running processes (snapshot) ps aux > /tmp/processes_$(date +%s).txt # Network connections netstat -tulpn > /tmp/netstat_$(date +%s).txt ss -tulpn > /tmp/ss_$(date +%s).txt # Open files lsof > /tmp/lsof_$(date +%s).txt # Scheduled tasks crontab -l > /tmp/crontab_$(date +%s).txt # Last logins last # System boot time uptime # Firewall rules iptables -L -n > /tmp/iptables_$(date +%s).txt
# Windows system information collection # System time systeminfo | findstr /C:"System Boot Time" # Logged-in users query user # Running processes tasklist > processes_$(date /t)_$(time /t).txt # Network connections netstat -ano > netstat_$(date /t)_$(time /t).txt # Open files openfiles > openfiles_$(date /t)_$(time /t).txt # Scheduled tasks schtasks /query /fo LIST > tasks_$(date /t)_$(time /t).txt # Event logs wevtutil epl Security C:\security-export.evtx wevtutil epl System C:\system-export.evtx

Phase 3: Containment, Eradication, Recovery

This phase stops attacker, removes all presence, and restores normal operations.

Containment Strategy

Immediate actions to stop incident from spreading:

  • System isolation: Disconnect affected systems from network (unplug cable, disable network interface)
  • Account revocation: Revoke credentials of affected accounts
  • Network segmentation: Isolate affected network segments
  • Block attacker IPs: Block known malicious IPs at firewall
  • Change passwords: Force password resets for affected users
  • Disable services: Disable services being exploited

Eradication

Remove attacker presence and vulnerabilities:

  • Malware removal: Identify and remove all malware, backdoors, webshells
  • Account cleanup: Remove attacker-created accounts, audit all accounts for unauthorized access
  • Backdoor identification: Check for scheduled tasks, persistence mechanisms, unauthorized services
  • Vulnerability patching: Patch vulnerabilities attacker exploited
  • Configuration review: Review and harden configurations (firewall rules, permissions, services)

Recovery

Restore normal operations:

  • System restoration: Restore from known-good backups created before incident
  • Configuration verification: Confirm restored systems match secure configuration baselines
  • Monitoring: Enhanced monitoring post-recovery to detect attacker return
  • Credential reset: Issue new credentials to all potentially affected users
  • Phased restoration: Restore critical systems first, monitor before restoring remaining systems

Phase 4: Post-Incident Activity

After incident resolved, perform thorough review and improve preparedness for future incidents.

Lessons Learned Meeting

  • What happened? (timeline, scope, impact)
  • How did we detect it? (detection effectiveness)
  • How did we respond? (what worked, what didn't)
  • What contributed to the incident? (root cause)
  • What could have prevented it?
  • What should we change?

Documentation

  • Incident report: Detailed incident documentation for organizational learning
  • Timeline: Precise incident timeline (detection, actions, milestones)
  • Evidence summary: Collected evidence and forensic findings
  • Financial impact: Quantified costs (response, remediation, lost business, legal)
  • Procedural updates: Update incident response plan and security policies

Digital Forensics Fundamentals

Digital forensics is scientific investigation and analysis of digital evidence. Forensic analysis determines what attackers did, how they gained access, and what data was accessed or stolen.

Forensic Principles

ACPO Good Practice Guide Principles (UK)
1. Actions taken to secure and audit exhibits should not change data 2. Person accessing exhibits should be competent and able to explain their actions 3. Audit trail or record of all processes applied to exhibits must be created 4. Person in charge of investigation has overall responsibility for ensuring law and principles adhered to

Chain of Custody

Chain of custody documents who handled evidence, when, and for what purpose. Preserves evidentiary integrity for legal purposes.

  • Documentation: Every access to evidence logged (who, when, purpose)
  • Sealing: Evidence sealed in tamper-evident bag with documentation
  • Hashing: Cryptographic hash calculated at each stage to verify integrity
  • Storage: Evidence stored in secure, access-controlled environment
Critical: Evidence Integrity
Evidence must be preserved in original, unmodified state. Never analyze evidence directly - always create forensic image (bit-for-bit copy) and analyze the copy. Any modification to evidence renders it inadmissible in court.

Disk Forensics

Disk forensics examines data stored on hard drives and solid state drives to reconstruct user activity and system state.

Forensic Disk Imaging

# Linux: Create disk image with dd (ensure no changes to evidence) dd if=/dev/sdb of=/dev/evidence/sdb_image.dd bs=4M conv=noerror,sync status=progress # Verify image with hash sha256sum /dev/evidence/sdb_image.dd > /dev/evidence/sdb_image.dd.sha256 # Calculate hash of original for comparison sha256sum /dev/sdb

dd copies every bit including free space and deleted files, preserving all evidence.

What Disk Forensics Reveals

  • Deleted files: Files that user deleted may be recoverable from free space
  • File timestamps: MAC times (Modified, Accessed, Created) reveal file access patterns
  • Disk artifacts: Prefetch files, browser cache, application logs, registry (Windows), .bash_history (Linux)
  • Suspicious files: Malware, backdoors, stolen data archives
  • Tools used: Installed applications, recently opened documents

Forensic Tools

  • Autopsy: Open-source disk forensics suite (GUI, timeline, keyword search)
  • Sleuth Kit: Command-line toolkit underlying Autopsy
  • EnCase: Commercial forensics suite with extensive capabilities
  • FTK Imager: Disk imaging and basic analysis
  • Volatility: Memory forensics (see next section)

Memory Forensics

Memory forensics analyzes RAM to find processes, network connections, and artifacts that do not persist to disk (fileless malware, encryption keys, injected code).

# Capture physical memory dump (Windows - requires admin privileges) # Using FTK Imager (GUI tool) # Or using Microsoft DumpIt utility # Linux: Capture memory cat /dev/mem > memory_dump.raw # Or using LiME (Linux Memory Extractor) insmod lime.ko "path=/evidence/memory_dump.raw format=lime"

Memory Forensics with Volatility

# Determine Windows profile volatility -f memory_dump.raw imageinfo # List running processes volatility -f memory_dump.raw --profile=Win7SP1x64 pslist # Process tree with parent/child relationships volatility -f memory_dump.raw --profile=Win7SP1x64 pstree # Network connections volatility -f memory_dump.raw --profile=Win7SP1x64 netscan # Console output (what user typed) volatility -f memory_dump.raw --profile=Win7SP1x64 consoles # Injected DLLs volatility -f memory_dump.raw --profile=Win7SP1x64 dlllist # Memory dump of specific process (analyze malware) volatility -f memory_dump.raw --profile=Win7SP1x64 memdump -p [PID] -D /evidence/

Memory forensics reveals evidence that never touches disk: fileless malware, injected code, encryption keys, clipboard data, unsaved documents.

Network Forensics

Network forensics examines network traffic to reconstruct communications, identify attack vectors, and trace attacker activities.

Packet Capture

# Capture packets with tcpdump (Linux) tcpdump -i eth0 -w capture.pcap # Capture specific network tcpdump -i eth0 -w capture.pcap net 192.168.1.0/24 # Capture specific port tcpdump -i eth0 -w capture.pcap port 443 # Windows: Packet capture with Wireshark (GUI tool)

Packet Analysis with Wireshark

  • Follow TCP stream: Reconstruct full HTTP, FTP, or other TCP conversations
  • Filtering: Apply filters to isolate relevant traffic (e.g., "http.request.method == 'POST'")
  • Payload extraction: Extract files from pcap (files transmitted over HTTP, FTP)
  • TLS analysis: TLS details if keys available, confirm encryption, certificate validation
  • Conversation statistics: "Conversations" summary shows who talked to whom

Netflow Analysis

Netflow records summarize network communications for large-scale analysis:

  • Source and destination IP
  • Source and destination port
  • Protocol (TCP, UDP, ICMP)
  • Packet and byte counts
  • Flow start and end timestamp

Netflow enables detection of:

  • Data exfiltration (unusually large outbound transfers)
  • Suspicious connections (long-duration connections from unexpected IPs)
  • Beaconing (regular connections to C2 servers)

Log Analysis

Event logs from systems, applications, network devices, and security tools reconstruct incident timeline and attacker activities.

Important Log Sources

  • System logs: /var/log/auth.log (Linux), Windows Event Log (Security, System, Application)
  • Application logs: Web server logs (Apache, Nginx), application logs (database, custom apps)
  • Network device logs: Firewall logs, router logs, switch logs, VPN logs
  • Security tools: IDS/IPS alerts, EDR alerts, antivirus logs
  • DNS logs: DNS query logs (useful for identifying C2 domain names)

Log Collection

# Linux logs /var/log/auth.log # Authentication events (ssh, sudo) /var/log/syslog # System messages /var/log/kern.log # Kernel messages ~/.bash_history # User command history ~/.zsh_history # Windows logs (via GUI: Event Viewer) Event Viewer → Windows Logs: - Security (logon events, privilege changes) - System (system events, service changes) - Application (application errors, events)

Timeline Construction

Forensic timeline reconstructs incident chronology. Timeline enables understanding of attack progression.

  • Sources: Combine evidence from disk, memory, logs, network captures
  • Timeline types: MAC times (file), event timestamps (logs), packet timestamps (network)
  • Analysis: Identify initial access, lateral movement, persistence mechanisms, exfiltration
  • Tools: Autopsy timeline, Plaso (log2timeline), custom scripts
Timeline Indicators
Initial access: First suspicious log entry from attacker IP Lateral movement: Access attempts to additional systems Persistence: Scheduled tasks, startup items, registry modifications Exfiltration: Large outbound data transfers Incident scope: Timeline reveals how long attacker had access and extent of compromise

Malware Analysis Basics

Malware analysis determines malware capabilities, behavior, and methods. Analysis enables understanding what attackers achieved and how to prevent similar threats.

Static Analysis

Static analysis examines malware without executing it.

  • File hashing: Calculate SHA256 - compare to threat intelligence databases (VirusTotal)
  • Strings extraction: Extract readable strings (URLs, IPs, file paths, commands)
  • Packing detection: Identify if malware is packed/obfuscated
  • Disassembly: Examine assembly code (requires reverse engineering expertise)
  • Import and export analysis: Examine DLL imports (functions malware uses), exports
# Static analysis tools # Calculate hash sha256sum malware.exe > malware.sha256 # Extract strings strings malware.exe > malware_strings.txt # Analyze ELF file (Linux) readelf -h malware_elf # Analyze PE file (Windows) objdump -p malware.exe pefile (Python library) # Check against VirusTotal python -m virustotal.malware_scanner malware.exe

Dynamic Analysis

Dynamic analysis executes malware in controlled environment to observe behavior.

  • Sandbox: Execute malware in isolated virtual machine (Cuckoo Sandbox, Remnux)
  • Network monitoring: Capture communications with C2 servers
  • Process monitoring: Observe spawned processes, registry modifications, file system changes
  • Screenshot capture: Periodic screenshots observe malware GUI
  • Disk changes: Track file creation, deletion, modification
Critical: Isolated Environment Required
Never execute suspicious malware on production systems or non-isolated environments. Malware can spread across networks, damage systems, and exfiltrate data. Use dedicated malware analysis sandbox (isolated virtual machine) that can be destroyed and restored after analysis.

Security Information and Event Management (SIEM)

SIEM systems collect, correlate, and analyze logs from across IT infrastructure. Effective SIEM implementation enables rapid incident detection and response.

SIEM Components

  • Log collection: Ingest logs from systems, applications, network devices, security tools via agents or syslog
  • Parsing and normalization: Convert logs to common format for analysis
  • Correlation: Identify related events across different sources
  • Alerting: Generate alerts based on rules and anomalies
  • Dashboard and reporting: Visualize security posture and trends

SIEM Implementation Best Practices

Log Sources

  • Endpoints: Workstations, servers (Windows Event Log, Linux syslog)
  • Network devices: Firewalls, routers, switches, VPN logs
  • Applications: Web servers, databases, custom applications
  • Directory services: Active Directory, LDAP
  • Security tools: IDS/IPS, EDR, antivirus, DLP
  • Identity systems: IAM, MFA logs

Alert Tuning

  • False positive reduction: Tune alerts to reduce noise
  • Threshold adjustment: Optimize thresholds (number of failed logins before alert)
  • Suppression rules: Suppress expected, non-malicious events
  • Severity classification: Classify alerts by severity for prioritization

Open Source SIEM

  • Wazuh: Open-source SIEM based on OSSEC, includes endpoint detection
  • ELK Stack (Elasticsearch, Logstash, Kibana): Flexible, widely used, requires configuration
  • Splunk: Commercial (has free version limited to daily volume)

Legal and Regulatory Reporting

Data breaches triggering legal reporting requirements differ by jurisdiction. Understanding obligations is critical for compliance.

US Regulations

  • State breach notification laws: All 50 states have laws requiring notification (typically 30-45 days)
  • HIPAA: Healthcare - breach affecting 500+ individuals requires notification
  • GLBA: Financial institutions - safeguard and security requirements
  • GRAMM-LEACH-BLILEY ACT: Safeguard customer records

GDPR (EU)

  • Article 33: Notify supervisory authority within 72 hours of becoming aware of breach
  • Article 34: Notify affected individuals without undue delay if high risk
  • Scope: Applies to EU citizens' data, regardless of organization location
  • Fines: Up to 4% of global annual revenue or 20 million EUR (whichever larger)

Working with Law Enforcement

  • Report incidents: Report cybercrime to local FBI field office (US) or appropriate agency
  • Preserve evidence: Law enforcement requires properly preserved evidence (forensic imaging, chain of custody)
  • Consult legal: Legal counsel should advise on law enforcement engagement
  • International cooperation: For trans-border attacks, engage appropriate international authorities

Common Incident Response Scenarios

Different incident types require specific response strategies.

Ransomware Response

  • Immediate action: Disconnect affected systems immediately (do not reboot)
  • Evidence collection: Capture memory, volatile data before shutdown
  • Determine scope: Identify all affected systems (ransomware encrypts network shares)
  • Backup verification: Confirm backup integrity before restoration
  • Decryption: Check No More Ransom project for decryption keys
  • Restore: Restore from verified backups after malware eradication
  • Payment decision: Consult legal and FBI - payment not recommended

Data Breach Response

  • Containment: Stop data exfiltration (block attacker IPs, revoke credentials)
  • Investigation: Determine what data was stolen, when, how many individuals affected
  • Legal notification: Consult legal regarding reporting obligations (GDPR, state laws)
  • Customer notification: Prepare clear, concise notification with remediation steps
  • Credit monitoring: Offer free credit monitoring to affected individuals
  • Public relations: Manage public communications honestly and transparently

Insider Threat Response

  • Evidence collection: Preserve logs and evidence without alerting employee
  • Legal consultation: Consult legal regarding employee rights and investigation procedures
  • HR coordination: HR handles employee communications and disciplinary actions
  • Access revocation: Revoke access when investigation confirms insider threat
  • Law enforcement: For malicious insiders (theft, sabotage), report to law enforcement

Business Continuity and Disaster Recovery

Business continuity planning ensures operations continue or rapidly resume after incident. Disaster recovery focuses on restoring IT systems.

Backup Strategy

3-2-1 Backup Rule

  • 3 copies of data: Original + 2 backups
  • 2 different storage media: E.g., disk + tape, or disk + cloud
  • 1 offsite backup: Backups stored in separate location (protects from physical disaster)

Backup Testing

  • Regular testing: Test backup restoration at least quarterly
  • Test full restoration: Don't just verify files exist - test complete system restoration
  • Document procedures: Document restoration procedures clearly and test them
  • RTO/RPO: Define Recovery Time Objective (how long to restore) and Recovery Point Objective (data loss acceptable)

Business Continuity Plan

  • Impact analysis: Identify critical systems and dependencies
  • Alternative procedures: Manual workarounds when systems unavailable
  • Communication plan: How to notify employees, customers, stakeholders
  • Roles and responsibilities: Who makes decisions, who executes procedures
  • Regular testing: Test business continuity plan annually
Critical: Test Backups Regularly
Backups that cannot be restored are worse than no backups. If you discover backup failure during incident response, you cannot recover. Test restoration regularly to ensure backups are valid and procedures work. Backup failures discovered during incident are catastrophic.
