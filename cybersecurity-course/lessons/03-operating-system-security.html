MODULE 3

Operating System Security

Introduction

Operating system security focuses on protecting the OS itself and the resources under its control. OS security mechanisms provide user authentication, access control, process isolation, and resource protection. Understanding both Linux and Windows hardening, patch management, and secure configuration is essential for securing any system.

Linux Security Fundamentals

Linux is built with multi-user security in mind from its Unix heritage. Understanding permissions, user accounts, and privileges is fundamental to Linux system administration and security.

Linux File Permissions

Linux uses a discretionary access control (DAC) model where each file has permission bits defining access for owner, group, and others.

Permission Bits
Each file has 9 permission bits + 3 special bits: Owner permissions (bits 1-3): - r (read): Allowed to read file contents or list directory - w (write): Allowed to modify file or create/delete files in directory - x (execute): Allowed to execute file or enter directory Group permissions (bits 4-6): - Same as owner, applied to group members Other permissions (bits 7-9): - Same as owner, applied to everyone else Special bits: - setuid (SUID): Execute with file owner's privileges - setgid (SGID): Execute with file group's privileges - sticky: Only owner can delete files in directory (critical for /tmp)
Example: Permission Bits
ls -l output: -rwxr-xr-- 1 alice staff 1234 Feb 21 10:30 script.sh Breakdown: - - (regular file type) - rwx (owner permissions: read, write, execute) - r-x (group permissions: read, execute only) - r-- (other permissions: read only) - alice (owner username) - staff (group name) - 1234 (file size in bytes) Octal representation: 754 r=4, w=2, x=1, so: - Owner: 4+2+1=7 - Group: 4+1=5 - Other: 4

chmod: Changing Permissions

# Octal mode chmod 755 script.sh # rwxr-xr-x chmod 644 file.txt # rw-r--r-- # Symbolic mode chmod u+x script.sh # Add execute for owner (user) chmod g-w file.txt # Remove write for group chmod o-r file.txt # Remove read for others chmod a+r file.txt # Add read for all (a=all)

chown: Changing Ownership

# Change owner chown bob file.txt # Now owned by bob # Change group chown :staff file.txt # Now group staff # Change both chown bob:staff file.txt # Owned by bob, group staff # Recursive change for directories chown -R bob:staff /home/bob/data

Special Bits: SUID, SGID, Sticky

SUID (Set User ID)

When executable with SUID bit runs, it executes with file owner's permissions, not user's permissions:

  • Purpose: Allow regular users to perform privileged operations (e.g., changing password requires writing /etc/shadow which is owned by root)
  • Representation: s in execute position (e.g., -rwsr-xr-x)
  • Risk: Vulnerable SUID binaries can be exploited for privilege escalation
# Set SUID bit chmod u+s /path/to/executable # Sets SUID bit chmod 4755 /path/to/executable # Same (4=100 in binary, SUID bit) # Find all SUID executables find / -perm -4000 2>/dev/null

SGID (Set Group ID)

Similar to SUID but for group permissions. On directories, causes new files to inherit directory's group:

  • Purpose: Collaborative directories (e.g., project directory where team members share group)
  • Directory: New files inherit group (useful for shared spaces)
  • Executables: Execute with file's group permissions
# Set SGID bit chmod g+s /shared/project # New files inherit group chmod 2775 /shared/project # Same (2=010 in binary, SGID bit)

Sticky Bit

On directories, only file owner can delete files, even if others have write permissions:

  • Purpose: Critical for world-writable directories (/tmp prevents users from deleting each other's files)
  • Representation: t in execute position (e.g., drwxrwxrwt)
# Set sticky bit chmod +t /shared/world_writable # Only owners can delete their files chmod 1777 /tmp # /tmp typically has sticky bit
Critical Permission Audit
Regularly audit SUID executables for unnecessary permissions: find / -perm -4000 -user root -type f 2>/dev/null If SUID binary for unnecessary reason, remove with: chmod u-s /path/to/executable Common vulnerable SUID binaries: old versions of sudo, screen, vim with unusual configurations.

Linux User and Privilege Management

Effective user management prevents unauthorized access, implements privilege separation, and ensures system accountability through proper access controls.

User and Group Management

# Create user (with home directory, copy skeleton files) useradd -m -s /bin/bash newuser # Set password passwd newuser # Delete user (remove home directory) userdel -r olduser # Create group groupadd developers # Add user to group usermod -a -G developers newuser # List user groups groups newuser # /etc/passwd format # username:password:UID:GID:comment:home:shell # x in password field means shadow file used

sudo: Superuser Privileges

sudo allows configured users to execute commands with elevated privileges (typically root) while providing audit trail.

# Edit sudoers (always use visudo, never edit directly) visudo # Common sudoers entries alice ALL=(ALL:ALL) ALL # Alice can execute all commands as any user bob ALL=(ALL) NOPASSWD: /usr/bin/systemctl # Bob can run systemctl without password %admin ALL=(ALL) ALL # All members of admin group # View sudo rules for current user sudo -l # Execute command as root sudo command_name # Execute as specific user sudo -u postgres psql # Execute as root with new environment sudo -i
sudo Security Best Practices
- Principle of least privilege: Grant only specific commands required, not ALL - Avoid NOPASSWD: Require password authentication unless automated - Audit log: sudo logs to /var/log/auth.log or /var/log/secure - Time-based: sudo ticket valid for 5 minutes by default (adjust via timestamp_timeout) - Limit sudo access: Create sudoers groups rather than individual users

Root Account

Critical Recommendation
Lock root account and rely entirely on sudo: sudo passwd -l root Rationale: - Direct root login increases attack surface (password-based, remote exploitation) - sudo provides audit trail (who ran what command when) - sudo allows fine-grained control (specific users, specific commands) - Compromised individual account less damaging than direct root compromise Emergency recovery: If sudo misconfigured, boot into single-user mode or use recovery media to modify sudoers.

Linux Hardening

Linux hardening involves configuring systems to minimize attack surface and resist common attacks. Many hardening measures can be applied during initial system setup and maintained through regular audits.

SSH Hardening

SSH is most common remote access method. Hardening SSH prevents brute force and unauthorized access.

# /etc/ssh/sshd_config # Disable root login PermitRootLogin no # Disable password authentication (use keys only) PasswordAuthentication no PubkeyAuthentication yes # Restrict to SSH protocol 2 Protocol 2 # Limit users allowed AllowUsers alice bob@specific_ip # Whitelist IP addresses (optional) AllowTcpForwarding no X11Forwarding no # Change default port (security through obscurity, but reduces automated attacks) Port 22222 # Restart SSH daemon sudo systemctl restart sshd
SSH Key Setup
Use SSH keys instead of passwords: # Generate ed25519 key (recommended) ssh-keygen -t ed25519 -C "user@example.com" # Copy public key to server ssh-copy-id user@server # Alternative (manual) ssh user@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys" < ~/.ssh/id_ed25519.pub Keys are stronger than passwords and resistant to brute force.

Firewall: UFW (Uncomplicated Firewall)

# Enable UFW sudo ufw enable # Allow SSH (do this before enabling firewall to avoid lockout) sudo ufw allow ssh # Or allow specific port (if SSH on non-default) sudo ufw allow 22222 # Allow HTTP and HTTPS sudo ufw allow 80/tcp sudo ufw allow 443/tcp # Deny all other traffic sudo ufw default deny incoming sudo ufw default allow outgoing # View status sudo ufw status verbose # Delete rule sudo ufw delete allow 80/tcp

Automatic Security Updates

# Install unattended-upgrades sudo apt install unattended-upgrades # Edit configuration sudo nano /etc/apt/apt.conf.d/50unattended-upgrades # Automatically install security updates Unattended-Upgrade::Allowed-Origins { "${distro_id}:${distro_codename}-security"; }; # Reboot automatically if required (set to "true" or "false") Unattended-Upgrade::AutoFixInterruptedDpkg "true"; Unattended-Upgrade::Automatic-Reboot "false"; # Enable service sudo systemctl enable unattended-upgrades sudo systemctl start unattended-upgrades

Fail2ban: Protect Against Brute Force

Fail2ban monitors log files and temporarily bans IP addresses showing malicious behavior (too many failed login attempts).

# Install fail2ban sudo apt install fail2ban # Create local configuration sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local # Edit jail.local for SSH protection [sshd] enabled = true port = ssh filter = sshd logpath = /var/log/auth.log maxretry = 5 findtime = 600 bantime = 3600 # Start and enable sudo systemctl enable fail2ban sudo systemctl start fail2ban # View status sudo fail2ban-client status sudo fail2ban-client status sshd

Disable Unused Services

# List all services systemctl list-units --type=service --state=running # Stop and disable service sudo systemctl stop service_name sudo systemctl disable service_name # Common services to disable if unnecessary sudo systemctl stop telnet # Unencrypted, insecure sudo systemctl disable telnet sudo systemctl stop rsh # Legacy, insecure # View listening ports sudo netstat -tlnp # Or use ss (modern replacement) sudo ss -tlnp

Windows Security Fundamentals

Windows Security Center (Windows Defender) provides integrated antivirus, firewall, and device security. Understanding Windows security mechanisms, policies, and hardening is essential for securing Windows systems.

Windows Access Control

Windows uses an access control list (ACL) model with Security Identifiers (SIDs) for users and groups.

Windows Security Entities
Security Identifiers (SIDs): - Unique identifier for each security principal (user, group, computer) - Examples: S-1-5-32-544 (Administrators), S-1-5-32-545 (Users), S-1-5-21-... (user SID) Access Control List (ACL): - List of Access Control Entries (ACEs) for resource Access Control Entry (ACE): - One permission entry (who, what rights, allow/deny) Permissions: - Read, Write, Execute, Delete, Full Control, Modify, etc.

File and Folder Permissions

# Via GUI Right-click file/folder → Properties → Security → Edit # Via PowerShell Get-Acl file.txt # View ACL Set-Acl file.txt $acl # Set ACL # Grant user "Alice" modify permission on folder $Path = "C:\Data" $User = "Alice" $Rights = "Modify" $Inheritance = "ContainerInherit, ObjectInherit" $acl = Get-Acl $Path $accessRule = New-Object System.Security.AccessControl.FileSystemAccessRule($User,$Rights,$Inheritance,"None","Allow") $acl.SetAccessRule($accessRule) Set-Acl $Path $acl

Default Windows Accounts

Account Purpose Recommendation
Administrator Built-in admin with full control Rename, strong password, limit use
Guest Limited guest access Disable, not needed in most environments
DefaultAccount System used for UAC Do not modify, disabled by default
Local accounts created Users created locally Remove if domain accounts used

Windows Hardening

Windows hardening reduces attack surface through configuration, security policies, and built-in security features.

User Account Control (UAC)

UAC prompts and consent mechanism allow standard users to perform administrative tasks by temporarily elevating privileges.

UAC Best Practices
- Do not disable UAC: Provides security boundary between admin user and standard user - Always notify: Highest security level - Admin Approval Mode: Admin users run as standard, prompt for elevation - LuaElevatedPrompt: Secure desktop for admin prompts Configuration via: Control Panel → User Accounts → Change User Account Control settings

Windows Defender Firewall

# Enable firewall (default enabled) Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled True # Block incoming connections by default Set-NetFirewallProfile -DefaultInboundAction Block # Allow specific application New-NetFirewallRule -DisplayName "Allow Remote Desktop" -Direction Inbound -Protocol TCP -LocalPort 3389 -Action Allow # Block specific port New-NetFirewallRule -DisplayName "Block Telnet" -Direction Inbound -Protocol TCP -LocalPort 23 -Action Block # View rules Get-NetFirewallRule | Format-List

Windows Updates

# Check update status via PowerShell Get-WindowsUpdate # Install updates Install-WindowsUpdate -AutoReboot # Configure automatic updates Set-WindowsUpdateAuto -Enable # View update history Get-HotFix | Sort-Object -Property InstalledOn -Descending

BitLocker: Full Disk Encryption

BitLocker encrypts entire drive, protecting data if device is lost or stolen.

# Enable BitLocker on C: drive (with TPM) Enable-BitLocker -MountPoint "C:" -EncryptionMethod XtsAes256 -UsedSpaceOnly -TpmProtector # Enable with password protector (add to TPM) Add-BitLockerKeyProtector -MountPoint "C:" -PasswordProtector # Backup recovery key to Active Directory Backup-BitLockerKeyProtector -MountPoint "C:" -RecoveryPasswordProtector # View BitLocker status Get-BitLockerVolume # Suspend BitLocker temporarily (e.g., for system update) Suspend-BitLocker -MountPoint "C:"
Critical: Backup Recovery Keys
BitLocker recovery keys are critical. If TPM fails or system changes, you must have recovery key to access encrypted drive. Backup recovery keys to: - Print and store securely - Save to USB drive in secure location - Store in password manager - Upload to cloud (encrypted)

Patch Management

Patch management is the process of acquiring, testing, and applying updates to software and operating systems. Regular patching addresses vulnerabilities and reduces attack surface.

Vulnerability Lifecycle

Timeline
1. Discovery: Researcher or attacker finds vulnerability 2. Responsible disclosure: Researcher reports to vendor (if ethical) 3. Patch development: Vendor develops and tests patch 4. Patch release: Vendor publishes update 5. Zero-day window: Time between disclosure and patch deployment when systems are vulnerable 6. Exploit development: Attackers develop exploits (sometimes before patch) 7. Wide deployment: Patches widely deployed, remaining unpatched systems increasingly vulnerable

Linux Patch Management

# Debian/Ubuntu sudo apt update # Update package lists sudo apt upgrade # Upgrade all packages sudo apt full-upgrade # Upgrade including dependencies sudo apt autoremove # Remove unnecessary packages # CentOS/RHEL sudo yum update # Update all packages sudo yum update package_name # Update specific package # View history sudo apt history # Ubuntu/Debian sudo yum history # CentOS/RHEL

Windows Patch Management

# Windows Update (GUI) Settings → Update & Security → Windows Update # Windows Update (PowerShell) Get-WindowsUpdate # View available updates Install-WindowsUpdate # Install updates Install-WindowsUpdate -AutoReboot # Install and reboot if needed # View update history Get-HotFix | Sort-Object InstalledOn -Descending # Configure automatic updates registry key: HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate\AU

Patch Management Best Practices

  • Test before deploying: Test patches in staging environment before production deployment
  • Prioritize critical patches: Deploy critical security patches first
  • Schedule regular updates: Set maintenance windows for patch deployment
  • Automate updates: Use automatic security updates where possible
  • Maintain backups: Always have recent backups before deploying patches
  • Monitor for issues: Watch for problems after updates
  • Document changes: Keep record of patches deployed and issues found
Emergency Patches
If zero-day vulnerability with active exploitation is discovered: - Prioritize testing for critical systems - Deploy emergency patch soon after limited testing - Prepare rollback plan - Monitor closely for issues - Document emergency deployment process

File System Security

File system security prevents unauthorized access, modification, or deletion of files. Proper permissions, encryption, and access controls protect data both at rest and in transit.

Linux File System Hardening

# Set umask (default permissions for new files) # Add to ~/.bashrc or /etc/profile umask 077 # New files: rw------- (600), directories: rwx------ (700) umask 022 # New files: rw-r--r-- (644), directories: rwxr-xr-x (755) # Verify current umask umask # Check for world-writable files find / -perm -0002 -type f 2>/dev/null # Check for SUID/SGID executables find / -perm -4000 -o -perm -2000 2>/dev/null

Linux Encryption: LUKS

LUKS (Linux Unified Key Setup) provides disk encryption. Encrypting disks protects data at rest.

# Encrypt partition (BEFORE use, destroys data) cryptsetup -y -v luksFormat /dev/sdb1 # Open encrypted partition cryptsetup luksOpen /dev/sdb1 encrypted_disk # Format with filesystem mkfs.ext4 /dev/mapper/encrypted_disk # Mount mount /dev/mapper/encrypted_disk /mnt/encrypted # On reboot, open and mount cryptsetup luksOpen /dev/sdb1 encrypted_disk mount /dev/mapper/encrypted_disk /mnt/encrypted # When done, unmount and close umount /mnt/encrypted cryptsetup luksClose encrypted_disk

# Encrypt individual file or folder (EFS - Encrypting File System) # Via GUI: Right-click → Properties → Advanced → Encrypt contents # Via PowerShell cipher /e C:\Data\secret.txt # Encrypt cipher /d C:\Data\secret.txt # Decrypt # Backup EFS certificate cipher /r # Generate backup to PFX file # BitLocker (see Windows hardening section) for full disk encryption

Process Security

Process security ensures isolation between processes, prevents privilege escalation, and limits damage from compromised processes.

Linux Process Isolation

  • User namespaces: Isolate user ID and group ID between processes (containers)
  • Mount namespaces: Isolate filesystem mount points
  • Network namespaces: Isolate network stacks
  • PID namespaces: Isolate process ID spaces
  • cgroups: Limit resource usage (CPU, memory, I/O)

Linux Containers (Docker)

# Docker security best practices # Run as non-root (add USER instruction in Dockerfile) USER appuser # Drop capabilities (remove unnecessary privileges) --cap-drop=ALL --cap-add=CAP_NET_BIND_SERVICE # Read-only filesystem --read-only # Resource limits --memory=512m --cpus=1 # Network isolation --network=none # No network # Volume mounts with read-only -v /host/path:/container/path:ro # Example secure container run docker run --name myapp \ --cap-drop=ALL \ --cap-add=CAP_NET_BIND_SERVICE \ --read-only \ --network host \ -v mydata:/data:ro \ myimage:latest

Windows Process Isolation

  • User Account Control: Elevation prompts
  • Windows Defender Application Control (WDAC): Restrict which applications can run
  • AppLocker: Application control policies
  • Windows Defender Credential Guard: Protect credentials in virtualization-based security
  • Sandboxing: Windows Sandbox for untrusted applications
