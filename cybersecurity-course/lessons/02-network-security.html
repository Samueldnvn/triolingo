MODULE 2

Network Security

Introduction

Network security involves protecting data during transmission, securing network infrastructure, and defending against network-based attacks. Understanding TCP/IP protocols, implementing firewalls and intrusion detection, and mitigating distributed denial-of-service attacks form the foundation of network security. This module covers the architecture of secure networks and the tools used to protect them.

TCP/IP Protocol Stack

The Internet Protocol Suite (TCP/IP) is the foundation of modern networking. Understanding its four-layer model is essential for identifying vulnerabilities, configuring security controls, and analyzing network traffic effectively.

The Four-Layer Model

OSI vs TCP/IP Comparison
TCP/IP Four Layers: 1. Network Interface Layer (Data Link + Physical) 2. Internet Layer (Network) 3. Transport Layer 4. Application Layer OSI Seven Layers (for reference): 7. Application Layer 6. Presentation Layer 5. Session Layer 4. Transport Layer 3. Network Layer 2. Data Link Layer 1. Physical Layer

IP Layer (Layer 2)

Internet Protocol (IP) handles packet routing and addressing across networks. Two versions are in use:

  • IPv4: 32-bit addresses, 4.3 billion possible addresses, largely exhausted
  • IPv6: 128-bit addresses, 340 undecillion possible addresses, future-proof

IP Security Considerations

  • No built-in authentication: Source IP addresses can be spoofed (IP spoofing attacks)
  • No encryption: IP packets are transmitted in plaintext unless wrapped by IPsec
  • No integrity checking: Packets can be modified in transit without detection
  • Asymmetric routing: Packets may take different paths, complicating monitoring

Transport Layer (Layer 3)

Two primary protocols operate at the transport layer: TCP and UDP, each with different security characteristics.

Transmission Control Protocol (TCP)

TCP provides reliable, ordered delivery through connection-oriented communication:

Three-Way Handshake: SYN → Server: "I want to connect" SYN-ACK ← Server: "I hear you, let's connect" ACK → Server: "Connection established"
  • Connection state: Maintains state for ongoing connections
  • Reliability: Retransmits lost packets, detects out-of-order packets
  • Flow control: Prevents overwhelming the receiver
  • Congestion control: Adjusts transmission rate based on network conditions

TCP Security Issues: The three-way handshake exposes TCP to SYN flood attacks and connection hijacking. State table exhaustion attacks overwhelm connection-tracking systems.

User Datagram Protocol (UDP)

UDP provides connectionless, unreliable datagram delivery:

  • No connection: Packets sent without establishing state
  • No reliability: Lost packets are not retransmitted
  • Faster: Lower overhead than TCP, suitable for real-time applications
  • Best-effort: No guarantees of delivery or order

UDP Security Issues: UDP is often used for amplification DDoS attacks because UDP responses can be much larger than requests. UDP flood attacks overwhelm services with connectionless packets.

Application Layer (Layer 4)

Applications use specific protocols at the application layer. Each protocol has unique security considerations:

Protocol Port Uses Common Vulnerabilities
HTTP 80 Web browsing Injection attacks, XSS, CSRF
HTTPS 443 Encrypted web Certificate spoofing, downgrade attacks
FTP 20, 21 File transfer Cleartext passwords, passive mode exploitation
SSH 22 Remote administration Weak keys, authentication bypass
SMTP 25 Email sending Email spoofing, spam relay
DNS 53 Domain resolution DNS cache poisoning, amplification attacks
RDP 3389 Remote desktop Brute force, BlueKeep vulnerability
Note
Port numbers 0-1023 are well-known ports (require root/admin privileges). Ports 1024-49151 are registered ports. Ports 49152-65535 are dynamic or private ports used for client-side connections.

Network Address Translation (NAT)

NAT allows multiple devices to share a single public IP address, translating private IP addresses to public ones for internet communication. While NAT provides some security by hiding internal IP addresses, it was designed to address IPv4 address exhaustion, not security.

Private IP Address Ranges (RFC 1918)

  • 10.0.0.0 - 10.255.255.255 (10.0.0.0/8)
  • 172.16.0.0 - 172.31.255.255 (172.16.0.0/12)
  • 192.168.0.0 - 192.168.255.255 (192.168.0.0/16)
Important
NAT is not a firewall. While NAT complicates direct inbound connections, it can be bypassed through UPnP, port forwarding, or application-layer protocols that establish connections from internal networks. Firewalls should be configured independently, not assumed from NAT.

Firewalls

Firewalls are network security devices that monitor and control incoming and outgoing traffic based on predetermined security rules. They act as the first line of defense, filtering malicious traffic and preventing unauthorized access.

Types of Firewalls

Packet-Filtering Firewalls (Stateless)

Stateless firewalls examine individual packets independently, making decisions based on header information:

  • Source IP and destination IP
  • Source port and destination port
  • Protocol (TCP, UDP, ICMP)
Example: Stateless Firewall Rule
Allow: TCP from source 192.168.1.10 to destination ANY on port 443 Deny: ALL from source 10.0.0.0/8 The firewall examines each packet and applies rules independently, with no memory of previous packets.

Limitations: Stateless firewalls cannot distinguish between response traffic and new connections because they do not track connection state. They also cannot detect application-layer attacks hidden within permitted traffic.

Stateful Inspection Firewalls

Stateful firewalls maintain connection state tables to track ongoing connections:

  • State tracking: Remembers connection details (source, destination, ports, sequence numbers)
  • Dynamic rules: Allows response traffic only for established connections
  • Context-aware: Understands TCP handshake, UDP flows, and ICMP associations
State Table Example
A connection is established when firewall sees SYN packet from internal host to external server. The firewall adds entry to state table: - Source: 192.168.1.10:45123 - Destination: 93.184.216.34:443 - State: ESTABLISHED Response packets (SYN-ACK, ACK) are automatically allowed because they match state table entry. If attacker tries to send packets to 192.168.1.10:45123 from 93.184.216.34 without prior SYN, packets are dropped because no matching state entry exists.

Advantages: Stateful firewalls provide stronger security by automatically allowing only response traffic for authorized connections and blocking unsolicited inbound traffic.

Next-Generation Firewalls (NGFW)

Next-generation firewalls add application-layer awareness:

  • Application identification: Recognizes applications, not just ports (e.g., distinguishes Facebook web chat from HTTP on port 443)
  • Deep packet inspection (DPI): Examines payload content for threats, signatures, and anomalies
  • IDPS integration: Intrusion detection and prevention built-in
  • User identity: Can enforce policies by user or group, not just IP address
  • SSL/TLS inspection: Decrypts and inspects encrypted traffic for threats
Critical Consideration: TLS Inspection
Decrypting TLS traffic for inspection requires the firewall to act as a man-in-the-middle using its own certificates. This introduces: - Certificate trust concerns (systems must trust firewall CA) - Privacy considerations (firewall sees decrypted traffic) - Performance overhead (decryption is computationally expensive) Balance security needs with privacy and performance. Not all SSL/TLS traffic should necessarily be decrypted.

Firewall Rule Management

Effective firewall configuration requires careful rule design:

  • Default deny: Start by denying all traffic, then explicitly allow what is needed
  • Specificity: Make rules as specific as possible to reduce attack surface
  • Rule ordering: Firewall policies are evaluated top-down; first match wins
  • Rule simplification: Avoid unnecessary rules, consolidate when possible
  • Auditing: Regularly review and remove unused rules
  • Logging: Log dropped traffic to detect attacks and misconfigurations
Example: Firewall Rule Ordering
1. Allow: TCP 192.168.1.10 → ANY:443 (This user's web access) 2. Deny: TCP ANY → 192.168.1.0/24:80 (Block inbound HTTP from anywhere) 3. Allow: TCP 10.0.0.0/8 → 192.168.1.0/24:22 (SSH from corporate network) 4. Deny: ALL (Drop everything else) If a packet from 10.0.0.50 to 192.168.1.10:22 arrives: - Rule 1: Does not match (not 443) - Rule 2: Does not match (to port 22, not 80) - Rule 3: MATCH! Allow packet. - Rules 4 never evaluated because rule 3 matched first.

Network Segmentation and DMZ

Divide networks into segments to contain breaches and implement security zones:

Common Network Segments

  • DMZ (Demilitarized Zone): Public-facing servers (web, email, VPN) isolated from internal network
  • Internal network: Production workstations and servers
  • Server network: Database and application servers
  • Guest network: Completely isolated for visitors
  • Management network: Administrators only, separate from production
DMZ Architecture
Internet → External Firewall → DMZ (web servers) → Internal Firewall → Internal Network If DMZ server is compromised, attacker cannot easily reach internal network because internal firewall blocks DMZ-to-internal traffic. DMZ servers cannot initiate connections to internal network except through tightly controlled rules.

Intrusion Detection and Prevention Systems

Intrusion Detection Systems (IDS) monitor network traffic or system activities to detect signs of malicious behavior. Intrusion Prevention Systems (IPS) go a step further by actively blocking detected threats.

Detection Methods

Signature-Based Detection

Compares traffic and patterns against known threat signatures:

  • Database-driven: Maintains signatures of known attacks
  • Pattern matching: Matches packets against signature patterns
  • Low false positives: Highly accurate for known threats
  • Misses new threats: Cannot detect zero-day attacks without existing signatures
Example Signature
Alert: TCP packet contains string "SELECT * FROM users WHERE password = '" This matches SQL injection attack pattern in HTTP payload. Signature indicates attacker attempting SQL injection against web application.

Anomaly-Based Detection

Detects deviations from baseline behavior:

  • Baseline learning: Learns normal traffic patterns over time
  • Statistical analysis: Detects statistically improbable deviations
  • Unknown threats: Can detect novel attacks without prior signatures
  • Higher false positives: Legitimate changes can trigger alerts
Anomaly Detection Examples
1. Traffic volume: Sudden spike from IP address scanning many ports 2. Protocol distribution: ICMP traffic appears when baseline shows only TCP/UDP 3. Time patterns: Activity at 3 AM network is normally quiet 4. Data exfiltration: Large outbound transfers when normal pattern is inbound-heavy Each deviation produces alert; security team investigates legitimate vs malicious.

Heuristic-Based Detection

Uses rules and logic to identify suspicious behavior:

  • Behavioral rules: "If X happens, it's likely malicious"
  • Expert systems: Encodes security analyst knowledge
  • Faster than anomaly: No learning period required
  • Better than pure signature: Can detect variations of known attacks

IDS vs IPS

Aspect IDS IPS
Action Alert only, does not block Actively blocks threats
Latency Minimal (can be out-of-band) Higher (inline processing)
Risk Risk of missing attacks Risk of false positives blocking legitimate traffic
Placement Can mirror traffic anywhere Must be inline (on network path)
Guideline
Use IPS in protect mode only after thoroughly tuning and validating rules in IDS (detect) mode. False positives in IPS can block legitimate business traffic. Start in detect mode, analyze alerts, and enable blocking for validated high-confidence rules.

Open Source IDPS Tools

  • Snort: Widely used signature-based IDS/IPS
  • Suricata: Multi-threaded IDS/IPS with application-layer detection
  • OSSEC: Host-based intrusion detection (system logs)
  • Wazuh: Security monitoring platform based on OSSEC
  • Zeek: Network security monitoring with deep protocol analysis

DDoS Attacks and Mitigation

Distributed Denial-of-Service attacks aim to overwhelm resources with traffic, rendering services unavailable. Attackers use botnets (networks of compromised computers) to coordinate attacks, making them difficult to block.

Types of DDoS Attacks

Volumetric Attacks

Overwhelms network bandwidth with massive traffic volume:

  • UDP floods: Large UDP packets flood target, saturating bandwidth
  • ICMP floods: Ping floods and ping of death attacks
  • Amplification attacks: Exploit services that return larger responses than requests
Example: DNS Amplification
Attacker sends DNS query "*" (request all records) to DNS resolver with spoofed source IP (victim's IP). Source IP: 203.0.113.50 (victim) Destination IP: 8.8.8.8 (DNS resolver) Query: 60 bytes DNS resolver responds to victim: Source IP: 8.8.8.8 Destination IP: 203.0.113.50 (victim) Response: 4000 bytes Amplification: 67x amplification (4000 / 60). Attacker sends queries to thousands of DNS resolvers, all responding to victim, overwhelming victim with 100+ Gbps traffic.

Protocol Attacks

Exploits weaknesses in network protocols and infrastructure:

  • SYN floods: Sends many TCP SYN packets without completing handshake, exhausting connection tables
  • ACK floods: Overwhelms system with ACK responses
  • Fragmentation attacks: Sends malformed packet fragments that crash systems
SYN Flood Mechanism
Attacker sends thousands of fake SYN packets with spoofed source IPs. Victim: SYN (IP: random non-existent IP) Victim's firewall: SYN-ACK sent to fake IP (no response) Wait state: 75 seconds (typical) before timeout Do this 1,000,000 times: 1,000,000 connection table entries remain in state. Result: Connection table full, legitimate SYN packets dropped because no space available

Application Layer Attacks

Targets specific application vulnerabilities:

  • HTTP floods: Overwhelms web server with HTTP requests
  • Slowloris: Sends HTTP requests slowly, keeping connections open to exhaust connection limits
  • Query-intensive attacks: Sends expensive queries (complex searches, large reports) to exhaust database/database connection pools

DDoS Mitigation Strategies

1. Network-Based Mitigations

  • Bandwidth over-provisioning: Have excess capacity to absorb attacks (costly)
  • Scrubbing centers: Route traffic through DDoS mitigation services that filter malicious traffic
  • Anycast routing: Distribute traffic across multiple geographic locations, distributing attack load
  • Rate limiting: Limit requests per IP address, per minute/second

2. Infrastructure Configurations

  • SYN cookies: Cryptographically encode TCP connection info in SYN-ACK, avoiding state table entries
  • Connection timeouts: Reduce SYN timeout from 75s to 5-10s, allowing faster recovery
  • Drop invalid packets: Drop packets with invalid headers, checksums, or source IPs (ingress filtering)
  • UDP rate limiting: Rate-limit responses to limit amplification attacks

3. Application-Level Mitigations

  • Web application firewalls (WAF): Filter HTTP-based attacks and flood attempts
  • Caching: Use CDNs to cache content, reducing load on origin
  • Load balancing: Distribute traffic across multiple servers, distributing attack load
  • Resource limits: Limit query complexity, timeouts, and connections per user

4. DDoS Mitigation Services

  • Cloudflare: Global CDN with automatic DDoS mitigation
  • AWS Shield / Cloudflare / Akamai: Managed services
  • Akamai Prolexic: Large-scale mitigation services
  • Imperva: Comprehensive DDoS protection
Critical: No Single Solution
No single DDoS mitigation technique handles all attacks. Defense in depth is required: leverage CDN caching, configure infrastructure hardening (SYN cookies, timeouts), rate limiting, and have DDoS mitigation service on standby. Plan responses before attacks occur.

Virtual Private Networks

Virtual Private Networks create encrypted tunnels over public networks, allowing secure remote access to private networks. VPNs are essential for remote workers, connecting branch offices, and securing public WiFi usage.

VPN Protocols

IPsec VPN

IPsec operates at the network layer (layer 3), providing strong encryption and authentication:

  • Two modes: Transport mode (IPsec in transport) and Tunnel mode (IPsec in tunnel)
  • Two protocols: AH (Authentication Header) and ESP (Encapsulating Security Payload)
  • Key exchange: IKE (Internet Key Exchange) for secure key negotiation
  • Strong encryption: Supports AES, AES-GCM, and modern ciphers
AH vs ESP
AH (Authentication Header): - Provides integrity and authentication - Does not encrypt payload (data visible) - Rarely used due to lack of encryption ESP (Encapsulating Security Payload): - Provides integrity, authentication, and encryption - Encrypts payload, making data unreadable - Most common IPsec protocol

WireGuard

Modern VPN protocol with improved simplicity and performance:

  • Cryptographically sound: Only uses modern, audited cryptography
  • Minimal codebase: 4,000 lines vs. 400,000+ for OpenVPN (easier to audit)
  • Performance: Kernel-space implementation, very fast throughput
  • No NAT traversal issues: Works through NAT with UDP hole punching
  • Simple configuration: Public keys only, no certificates hierarchy
Recommendation
For new deployments, prefer WireGuard over IPsec/OpenVPN. WireGuard offers better security (simpler, smaller codebase), easier setup, superior performance, and modern cryptography. However, IPsec remains useful for network-to-network (site-to-site) VPNs where broad device compatibility is required.

L2TP/IPsec

Layer 2 Tunneling Protocol combined with IPsec encryption:

  • L2TP: Provides tunneling (encapsulation)
  • IPsec: Provides encryption (ESP)
  • Broadly supported: Works on Windows, macOS, Linux, iOS, Android out-of-box
  • Performance overhead: Double encapsulation, higher overhead than pure IPsec or WireGuard

VPN Security Hardening

  • Strong authentication: Use certificates rather than pre-shared keys where possible
  • Cryptographic best practices: Modern ciphers (AES-256-GCM, ChaCha20-Poly1305), no outdated algorithms (3DES, MD5, SHA1)
  • MFA (Multi-Factor Authentication): Require MFA for VPN access
  • Least privilege: VPN users access only required resources, not entire network
  • Network segmentation: Place VPN users in dedicated zone, not directly into internal network
  • Logging: Log VPN connections, users, and data transferred for audit and incident response
  • Regular updates: Keep VPN software patched for vulnerabilities
Site-to-Site vs Remote Access VPN
Site-to-Site VPN: Connects two networks (e.g., branch office to headquarters) - Network-to-network tunnel - Gateways (routers/firewalls) handle VPN Remote Access VPN: Allows individual users to access corporate network remotely - User-to-network tunnel - Client software on user laptop connects to VPN gateway Use cases differ: Site-to-site for permanent connections between locations; Remote access for individual remote workers.

Common VPN Vulnerabilities

  • Weak credentials: Brute force attacks on VPN accounts
  • Pre-shared key compromise: PSKs stored in insecure locations or shared inappropriately
  • NAT traversal issues: Misconfiguration blocking VPN through NAT
  • Outdated protocols: Using deprecated protocols (PPTP, L2F, L2TP without IPsec)
  • Cipher downgrade attacks: Allowing negotiation to weak ciphers for compatibility
  • VPN concentrator compromise: Attack on VPN gateway allows network access
Critical: Avoid PPTP
PPTP (Point-to-Point Tunneling Protocol) is fundamentally broken: - Encryption can be cracked in hours (MS-CHAPv2 weaknesses) - Protocol vulnerabilities known since 1999 - Microsoft officially deprecated PPTP in 2012 PPTP provides no real security. Use IPsec, WireGuard, or OpenVPN instead.
