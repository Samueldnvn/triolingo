IoT Connectivity & Networking Fundamentals

Understanding Communication Protocols for IoT Devices

Connectivity is the backbone of any IoT system. Choosing the right networking technology determines power consumption, range, bandwidth, and overall system architecture. This module explores the major connectivity options for IoT devices and when to use each.

CONNECTIVITY DECISION FACTORS
  • Range: Distance between devices (meters to kilometers)
  • Power Consumption: Battery life implications
  • Bandwidth: Data throughput requirements
  • Latency: Communication delay sensitivity
  • Cost: Hardware and infrastructure expenses
  • Topology: Network structure (star, mesh, hybrid)

IoT Connectivity Spectrum

Technology Range Bandwidth Power Best For
WiFi ~50m High (Mbps-Gbps) High High-bandwidth devices, indoor gateways
Bluetooth LE ~100m Low (Kbps) Very Low Short-range sensors, wearables
Zigbee ~100m (mesh extends) Low (250 Kbps) Low Home automation, sensor networks
LoRaWAN 2-15km Very Low (0.3-50 Kbps) Very Low Long-range sensors, agriculture
Cellular (NB-IoT) Global Low (250 Kbps) Low-Medium Wide-area tracking, utilities
EXAMPLE: Smart Home Connectivity Mix

A typical smart home uses multiple connectivity technologies: WiFi for cameras and thermostats (high bandwidth), Zigbee for smart lights and sensors (mesh network), and Bluetooth LE for mobile phone connections to individual devices. A central hub bridges these networks to the internet.

PATH LOSS MODEL
Free-space path loss (in dB): FSPL = 20 log₁₀(d) + 20 log₁₀(f) + 32.44
where d = distance in km, f = frequency in MHz

WiFi for IoT

High-Bandwidth Connectivity for IoT Devices

WiFi (IEEE 802.11) is the most widely available connectivity option for IoT devices due to existing infrastructure. While power-hungry compared to alternatives, WiFi offers high bandwidth and broad device compatibility, making it ideal for IoT gateways and devices that can be mains-powered.

WiFi Standards for IoT

WIFI STANDARDS COMPARISON
  • 802.11b/g/n (2.4GHz): Longest range, most interference Legacy but still common
  • 802.11ac/n/a (5GHz): Shorter range, less interference, higher bandwidth
  • 802.11ax (WiFi 6): Improved efficiency, better power management for IoT
  • 802.11ah (HaLow): Sub-1GHz, long range, low power (emerging)

POWER CONSIDERATIONS

WiFi consumes 100-500mA active vs 1-10mA for BLE. For battery-powered devices, WiFi requires careful power management: sleep modes, duty cycling, and wakeup scheduling are essential. Consider ESP32's deep sleep (0.1mA) vs active transmission (~150mA).

WiFi Security Best Practices

SECURITY REQUIREMENTS
  • Always use WPA2-AES or WPA3 encryption
  • Implement certificate-based device authentication
  • Use enterprise WiFi (WPA2-EAP) for critical deployments
  • Implement secure credential storage (never hardcode passwords)
  • Regularly rotate WiFi credentials and certificates
EXAMPLE: ESP32 WiFi Manager
public class WiFiManager {

Bluetooth Low Energy (BLE)

Ultra-Low Power Short-Range Communication

Bluetooth Low Energy, part of Bluetooth 4.0+, was specifically designed for IoT applications. With power consumption measured in microamps and support for coin-cell batteries, BLE is ideal for sensors, wearables, and other battery-operated devices requiring intermittent communication.

BLE KEY CHARACTERISTICS
  • Peak Current: 15mA (transmission) vs 40mA for classic Bluetooth
  • Sleep Current: ~1µA in deep sleep
  • Connection Time: <6ms (vs 100ms+ for classic)
  • Range: Up to 100m (class 2), up to 1km with long-range extension
  • Throughput: ~1 Mbps

BLE Architecture

BLE TERMINOLOGY
  • GATT (Generic Attribute Profile): Defines how data is transmitted
  • Service: Collection of characteristics (e.g., Heart Rate Service)
  • Characteristic: Data value with properties (read/write/notify)
  • Descriptor: Additional information about characteristics
  • Peripheral: Device that provides services (e.g., sensor)
  • Central: Device that connects to peripherals

import tinyb.BluetoothAdapter;
import tinyb.BluetoothDevice;
import tinyb.BluetoothGattService;

public class BLEPeripheral {
    public void startSensorService() {

        BluetoothGattService sensorService = new BluetoothGattService(
            UUID.fromString("0000180A-0000-1000-8000-00805F9B34FB"),
            BluetoothGattService.PRIMARY
        );
        
        // Add temperature characteristic
        BluetoothGattCharacteristic tempChar = new BluetoothGattCharacteristic(
            UUID.fromString("00002A6E-0000-1000-8000-00805F9B34FB"),

            BluetoothGattCharacteristic.NOTIFY
        );
        
        sensorService.addCharacteristic(tempChar);
        adapter.addService(sensorService);
    }
}
EXAMPLE: BLE Sensor Data Format
// BLE Heart Rate Monitoring Service (UUID 0x180D)
@Service
public class HeartRateService {
    @Characteristic(uuid = "0x2A37", properties = READ | NOTIFY)
    private short heartRateMeasurement;
    
    @Characteristic(uuid = "0x2A38", properties = READ)
    private byte bodySensorLocation;
    
    // Format: [flags(1B)][heartRateValue(1-2B)][rrInterval(2B)...]
    public byte[] getMeasurementBytes() {
        ByteBuffer buffer = ByteBuffer.allocate(3);
        buffer.put((byte) 0x01); // 8-bit heart rate format, no sensor contact
        buffer.put((byte) (heartRateMeasurement & 0xFF));
        return buffer.array();
    }
}
BLE SECURITY CONSIDERATIONS
  • Use LE Secure Connections for key exchange
  • Implement bonding to store encryption keys
  • Validate all incoming data (BLE packet injection attacks)
  • Use random addresses when privacy is required
  • Away mode: disable discoverability after pairing complete

Zigbee Mesh Networking

Low-Power, Self-Healing Sensor Networks

Zigbee is an IEEE 802.15.4-based specification for high-level communication protocols used to create personal area networks with small, low-power digital radios. Its mesh networking capability allows devices to relay data for each other, extending range and providing redundancy.

ZIGBEE KEY FEATURES
  • Mesh Topology: Multi-hop routing extends range automatically
  • Self-Healing: Network reconfigures when nodes fail
  • Low Power: Years of battery life for sensors
  • Device Roles: Coordinator, Router, End Device
  • Protocol Stack: Physical, MAC, Network (NWK), Application (APS)

Zigbee Device Types

Role Function Power Mains Required
Coordinator Forms network, trusts center Mains Powered Yes
Router Relays packets, extends mesh Mains Powered Yes
End Device Leaf node, no routing Battery No

Zigbee Applications Framework (ZCL)

COMMON ZIGBEE CLUSTERS
  • Basic (0x0000): Device identification, location
  • Power Configuration (0x0001): Battery status
  • On/Off (0x0006): Switch controls
  • Level Control (0x0008): Dimming
  • Temperature Measurement (0x0402): Sensors
  • Color Control (0x0300): RGB color
EXAMPLE: Zigbee Light Cluster Implementation
public class OnOffCluster implements ZigbeeCluster {
    private static final int CLUSTER_ID = 0x0006;
    private boolean state = false;
    
    public byte[] handleCommand(int commandId, byte[] payload) {
        switch (commandId) {
            case 0x00: // Off
                state = false;
                updateHardware(false);
                return new byte[] { (byte) commandId, 0x00 };
                
            case 0x01: // On
                state = true;
                updateHardware(true);
                return new byte[] { (byte) commandId, 0x00 };
                
            case 0x02: // Toggle
                state = !state;
                updateHardware(state);
                return new byte[] { (byte) commandId, 0x00 };
        }
        return null;
    }
}

Java with Zigbee4J

ZIGBEE COORDINATOR WITH ZIGBEE4J
import com.zsmartsystems.zigbee.ZigBeeNetwork;
import com.zsmartsystems.zigbee.ZigBeeNetworkManager;

public class ZigbeeCoordinator {
    private ZigBeeNetwork network;
    
    public void startNetwork() {
        ZigBeeNetworkManager manager = new ZigBeeNetworkManager(
            new SerialPort("/dev/ttyUSB0")
        );
        
        manager.setSerializer(new ZigBeeSerializerZigBeeDongle());
        network = manager.initialize();
        
        // Start discovery
        network.addDiscoveryListener(this::onDeviceDiscovered);
        network.startDiscovery();
    }
    
    private void onDeviceDiscovered(ZigBeeEndpoint device) {
        System.out.println("Device discovered: " + device.getEndpointId());
        System.out.println("Clusters: " + device.getInputClusters());
    }
}

LoRaWAN Long-Range IoT

Kilometer-Range Low-Power Wide Area Networking

LoRaWAN (Long Range Wide Area Network) enables long-range communication (up to 15 km in rural areas) at low power consumption. It's ideal for applications requiring periodic data transmission over wide areas: agriculture, smart cities, asset tracking, and environmental monitoring.

LORAWAN ARCHITECTURE
  • End Devices: Sensors, actuators (Class A/B/C)
  • Gateways: Forward packets to network server (star-of-stars)
  • Network Server: Manages devices, handles uplink/downlink
  • Application Server: Processes application data

LoRaWAN Device Classes

Class Downlink Power Use Best For
Class A After uplink only Lowest Battery sensors
Class B scheduled receive windows Low-Medium Controlled actuators
Class C Continuous listening Highest Mains-powered devices
SPREADING FACTOR TRADEOFFS
  • SF7: Fastest (293 bps), shortest range, least power
  • SF8: 176 bps, ~12 km range
  • SF9: 110 bps, ~14 km range
  • SF10: 61 bps, ~15 km range
  • SF11: 37 bps, ~18 km range
  • SF12: Slowest (18 bps), longest range (~22 km), most power

LoRaWAN with Java

LORAWAN CLASS A DEVICE WITH SEMTECH SDK
import com.semtech.lorawan.LoRaDevice;
import com.semtech.lorawan.LoRaPacket;

public class LoRaSensor {
    private LoRaDevice device;
    private byte[] appEUI;
    private byte[] appKey;
    
    public void sendSensorData(int temperature, int humidity) {
        // Prepare payload (temperature in °C, humidity in %)
        byte[] payload = new byte[2];
        payload[0] = (byte) (temperature & 0xFF);
        payload[1] = (byte) (humidity & 0xFF);
        
        LoRaPacket packet = new LoRaPacket();
        packet.setPayload(payload);
        packet.setPort(1); // Application port
        packet.setSpreadingFactor(10); // Balance range/speed
        
        // Transmit and switch to receive windows
        device.sendPacket(packet);
        device.openRX1Window(1000); // 1 second after TX
        device.openRX2Window(2000); // 2 seconds after TX
    }
    
    public void onDownlink(LoRaPacket packet) {
        byte[] fcntDown = packet.getFrameCounter();
        byte[] fopts = packet.getFOpts();
        byte[] payload = packet.getPayload();
        
        // Handle confirmed data or MAC commands
        System.out.println("Downlink received on port " + packet.getPort());
    }
}
EXAMPLE: CayenneLPP LoRaWAN Payload Format
public class CayenneLPP {
    public static byte[] encodeReading() {
        // Cayenne Low Power Payload format
        ByteBuffer buffer = ByteBuffer.allocate(11);
        
        // Digital Input (Channel 1)
        buffer.put((byte) 0x01);        // Channel
        buffer.put((byte) 0x00);        // Digital Input type
        buffer.put((byte) 0x01);        // Value (1 = on)
        
        // Analog Input (Channel 2)
        buffer.put((byte) 0x02);        // Channel
        buffer.put((byte) 0x02);        // Analog Input type, 16-bit
        buffer.putShort((short) 1234);  // Value
        
        // Temperature (Channel 3)
        buffer.put((byte) 0x03);        // Channel
        buffer.put((byte) 0x67);        // Temperature type, 16-bit, °C
        buffer.putShort((short) 2250);  // 22.50°C
        
        return buffer.array();
    }
}
LORAWAN DUTY CYCLE LIMITATIONS

LoRaWAN regulations limit transmission time: EU (1% duty cycle), US (90 seconds/day per channel). Class A devices must respect these limits. Adaptive data rate (ADR) automatically optimizes spreading factor and power to maximize network capacity.

Network Security

Securing IoT Communication Infrastructure

IoT networks present unique security challenges: low-power devices with limited compute, often deployed in physically accessible locations. Security must be designed from the ground up, not added as an afterthought.

IOT NETWORK SECURITY THREATS
  • Eavesdropping: Intercepting unencrypted communications
  • Man-in-the-Middle: Altering or relaying communications
  • Replay Attacks: Resending captured valid packets
  • Jamming: Disrupting wireless communications
  • Node Capture: Physical access compromised

Security Architecture Layers

DEFENSE IN DEPTH
  • Physical: Tamper detection, secure boot
  • Network: Encryption, authentication, firewalls
  • Application: Secure protocols, input validation
  • Cloud: Access controls, audit logging

Encryption Protocols

Protocol Key Size Overhead Best For
AES-128 128-bit Low Device encryption
TLS 1.3 256-bit High HTTPS, MQTT over TLS
DTLS 256-bit Medium CoAP, UDP-based
ChaCha20-Poly1305 256-bit Medium Mobile, constrained devices
SECURE MQTT CONNECTION WITH JAVA
import org.eclipse.paho.client.mqttv3.MqttClient;
import javax.net.ssl.SSLContext;
import java.security.KeyStore;

public class SecureMQTTClient {
    public MqttClient createSecureClient() throws Exception {
        // Load client certificate
        KeyStore clientStore = KeyStore.getInstance("PKCS12");
        clientStore.load(new FileInputStream("client.p12"), "password".toCharArray());
        
        // Configure SSL context
        SSLContext sslContext = SSLContext.getInstance("TLSv1.3");
        KeyManagerFactory kmf = KeyManagerFactory.getInstance(KeyManagerFactory.getDefaultAlgorithm());
        kmf.init(clientStore, "password".toCharArray());
        sslContext.init(kmf.getKeyManagers(), null, null);
        
        // Create MQTT client with SSL
        String brokerUrl = "ssl://mqtt.example.com:8883";
        MqttClient client = new MqttClient(brokerUrl, "iot-device-001");
        MqttConnectOptions options = new MqttConnectOptions();
        options.setSocketFactory(sslContext.getSocketFactory());
        options.setCleanSession(true);
        
        client.connect(options);
        return client;
    }
}
SECURE CREDENTIAL STORAGE
  • Never hardcode passwords or keys in source code
  • Use hardware security modules (HSM) when available
  • Encrypt credentials at rest with device-specific keys
  • Implement certificate rotation without re-provisioning
  • Secure boot ensures only signed firmware runs
