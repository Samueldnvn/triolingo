Communication Protocols

The Glue That Connects IoT Devices

IoT devices need to communicate with each other, with cloud services, and with user applications. Communication protocols define how this data is structured, transmitted, and received. Choosing the right protocol is critical for IoT application success.

KEY PROTOCOLS COVERED
  • HTTP/REST: Ubiquitous, easy to use, but sometimes heavyweight for IoT
  • MQTT: Lightweight publish/subscribe, ideal for constrained devices
  • CoAP: Designed for constrained nodes and networks (UDP-based)
  • WebSockets: Real-time bidirectional communication
PROTOCOL SELECTION GUIDE:
• Use HTTP for API-based IoT and when web infrastructure is available
• Use MQTT for IoT sensor networks with many devices
• Use CoAP for heavily constrained devices with unreliable networks
• Use WebSockets for real-time dashboards and user interfaces

HTTP/REST FOR IOT

What is HTTP?

HTTP (Hypertext Transfer Protocol) is the foundation of the World Wide Web. It's a request-response protocol where clients send requests to servers and receive responses. REST (Representational State Transfer) is an architectural style that uses HTTP for creating APIs.

HTTP Methods in IoT

// POST - Send sensor data to server
POST /api/sensors/temperature HTTP/1.1
Host: iot.example.com
Content-Type: application/json

{
    "deviceId": "sensor-001",
    "temperature": 23.5,
    "timestamp": "2026-02-18T21:52:00Z"
}

// GET - Retrieve device status
GET /api/devices/sensor-001 HTTP/1.1
Host: iot.example.com

// PUT - Update device configuration
PUT /api/devices/sensor-001/config HTTP/1.1
Host: iot.example.com
Content-Type: application/json

{
    "samplingRate": 5000,
    "enabled": true
}

// DELETE - Remove device registration
DELETE /api/devices/sensor-001 HTTP/1.1
Host: iot.example.com

Java HTTP Client Example

import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.time.Duration;
import java.util.concurrent.Executors;

// Simple IoT device sending temperature data
public class IotHttpClient {

    private static final String API_ENDPOINT = "https://iot.example.com/api/sensors/temperature";
    private static final String DEVICE_ID = "sensor-001";
    private static final HttpClient client = HttpClient.newBuilder()
        .version(HttpClient.Version.HTTP_2)
        .connectTimeout(Duration.ofSeconds(10))
        .executor(Executors.newSingleThreadExecutor())
        .build();

    public static void sendTemperature(double temperature) throws Exception {
        // Create JSON payload
        String jsonBody = String.format(
            "{\"deviceId\":\"%s\",\"temperature\":%.2f,\"timestamp\":\"%s\"}",
            DEVICE_ID,
            temperature,
            java.time.Instant.now().toString()
        );

        // Build HTTP request
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create(API_ENDPOINT))
            .header("Content-Type", "application/json")
            .header("Authorization", "Bearer YOUR_API_KEY")
            .POST(HttpRequest.BodyPublishers.ofString(jsonBody))
            .timeout(Duration.ofSeconds(5))
            .build();

        // Send request and get response
        HttpResponse response = client.send(
            request,
            HttpResponse.BodyHandlers.ofString()
        );

        System.out.println("Status: " + response.statusCode());
        System.out.println("Response: " + response.body());
    }

    public static void main(String[] args) {
        try {
            // Simulate reading temperature
            double temperature = readTemperatureSensor();
            sendTemperature(temperature);
        } catch (Exception e) {
            System.err.println("Error sending data: " + e.getMessage());
        }
    }

    private static double readTemperatureSensor() {
        // Simulated sensor reading
        return 22.5 + (Math.random() * 5);
    }
}

MQTT - MESSAGE QUEUING TELEMETRY TRANSPORT

What is MQTT?

MQTT is a lightweight, publish/subscribe messaging protocol designed for IoT and small sensors with high-latency or unreliable networks. It uses a broker to manage message distribution between clients.

MQTT Architecture

KEY CONCEPTS
  • Broker: Central server that receives and routes messages (e.g., Mosquitto, HiveMQ)
  • Publisher: Device that sends messages to topics
  • Subscriber: Client that receives messages from topics
  • Topic: Hierarchical string that defines message category (e.g., "sensors/livingroom/temperature")
  • QoS Levels: Quality of Service levels (0: fire and forget, 1: at least once, 2: exactly once)

MQTT Message Flow

Topic Structure (hierarchical):
sensors/livingroom/temperature
sensors/livingroom/humidity
sensors/kitchen/temperature
actuators/livingroom/light
actuators/livingroom/fan

Publisher (Sensor Device):
Subscribe: Not needed (only publishes)
Publish: sensors/livingroom/temperature → "23.5"

Subscriber (Cloud Service):
Subscribe: sensors/#
Publish: actuators/livingroom/fan → "ON" (to control fan)

Subscriber (Mobile App):
Subscribe: sensors/livingroom/#

Java MQTT Client Example

import org.eclipse.paho.client.mqttv3.*;
import org.eclipse.paho.client.mqttv3.persist.MemoryPersistence;

// IoT device publishing sensor data via MQTT
public class IotMqttPublisher {

    private static final String BROKER_URL = "tcp://broker.hivemq.com:1883";
    private static final String CLIENT_ID = "sensor-temperature-001";
    private static final String TOPIC = "sensors/room1/temperature";
    private static final int QOS = 1; // At least once
    private static final int INTERVAL_MS = 5000; // Send every 5 seconds

    private MqttClient client;

    public void connect() throws MqttException {
        // Create MQTT client
        client = new MqttClient(BROKER_URL, CLIENT_ID, new MemoryPersistence());

        // Configure connection options
        MqttConnectOptions options = new MqttConnectOptions();
        options.setCleanSession(true);
        options.setConnectionTimeout(30);
        options.setKeepAliveInterval(60);
        options.setAutomaticReconnect(true);

        // Connect to broker
        System.out.println("Connecting to MQTT broker...");
        client.connect(options);
        System.out.println("Connected!");

        // Set callback for connection_lost
        client.setCallback(new MqttCallback() {
            @Override
            public void connectionLost(Throwable cause) {
                System.err.println("Connection lost: " + cause.getMessage());
            }

            @Override
            public void messageArrived(String topic, MqttMessage message) {
                // Publishers don't typically receive messages, but can
                System.out.println("Received message on " + topic);
            }

            @Override
            public void deliveryComplete(IMqttDeliveryToken token) {
                try {
                    System.out.println("Message delivered: " + token.getMessageId());
                } catch (Exception e) {
                    e.printStackTrace();
                }
            }
        });
    }

    public void publishTemperature(double temperature) throws MqttException {
        String payload = String.format("{\"temp\":%.2f,\"ts\":\"%s\"}",
            temperature,
            java.time.Instant.now().toString()
        );

        MqttMessage message = new MqttMessage(payload.getBytes());
        message.setQos(QOS);
        message.setRetained(false);

        client.publish(TOPIC, message);
        System.out.println("Published to " + TOPIC + ": " + payload);
    }

    public void disconnect() throws MqttException {
        if (client.isConnected()) {
            client.disconnect();
            client.close();
            System.out.println("Disconnected");
        }
    }

    public static void main(String[] args) {
        IotMqttPublisher publisher = new IotMqttPublisher();
        try {
            publisher.connect();

            // Publish temperature every 5 seconds
            while (true) {
                double temp = 20.0 + (Math.random() * 10);
                publisher.publishTemperature(temp);
                Thread.sleep(INTERVAL_MS);
            }
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            try {
                publisher.disconnect();
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }
}

MQTT Subscriber Example

import org.eclipse.paho.client.mqttv3.*;

// Service that listens to sensor data
public class IotMqttSubscriber {

    private static final String BROKER_URL = "tcp://broker.hivemq.com:1883";
    private static final String CLIENT_ID = "subscriber-service-001";
    private static final String TOPIC = "sensors/#"; // All sensors
    private static final int QOS = 1;

    private MqttClient client;

    public void connectAndSubscribe() throws MqttException {
        client = new MqttClient(BROKER_URL, CLIENT_ID, new MemoryPersistence());

        MqttConnectOptions options = new MqttConnectOptions();
        options.setCleanSession(true);
        options.setConnectionTimeout(30);

        client.connect(options);

        // Subscribe to topics
        client.subscribe(TOPIC, QOS);
        System.out.println("Subscribed to: " + TOPIC);

        // Set callback to handle incoming messages
        client.setCallback(new MqttCallback() {
            @Override
            public void connectionLost(Throwable cause) {
                System.err.println("Connection lost");
            }

            @Override
            public void messageArrived(String topic, MqttMessage message) {
                String payload = new String(message.getPayload());
                System.out.println("Topic: " + topic);
                System.out.println("Message: " + payload);

                // Process sensor data
                if (topic.contains("temperature")) {
                    processTemperatureData(payload);
                }
            }

            @Override
            public void deliveryComplete(IMqttDeliveryToken token) {
                // Not applicable for subscribers
            }
        });
    }

    private void processTemperatureData(String json) {
        // Parse and store in database, trigger alerts, etc.
        System.out.println("Processing temperature data: " + json);
    }

    public static void main(String[] args) {
        IotMqttSubscriber subscriber = new IotMqttSubscriber();
        try {
            subscriber.connectAndSubscribe();

            // Keep running
            Thread.sleep(Long.MAX_VALUE);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
MQTT IS IDEAL FOR IoT BECAUSE:
• Very small header overhead (2 bytes minimum)
• Efficient for constrained devices
• Supports real-time communication
• One-to-many message distribution
• Works over unreliable networks
• Three QoS levels for reliability

COAP - CONSTRAINED APPLICATION PROTOCOL

What is CoAP?

CoAP (Constrained Application Protocol) is a specialized web transfer protocol for use with constrained nodes and networks in the IoT. It uses UDP instead of TCP, making it ideal for low-power, lossy networks.

CoAP vs HTTP Comparison

Feature HTTP CoAP
Transport TCP (reliable) UDP (lightweight)
Header Size ~800+ bytes ~4 bytes (minimum)
Methods GET, POST, PUT, DELETE, etc. GET, POST, PUT, DELETE
Reliability Guaranteed by TCP Optional confirmable messages
Security TLS/SSL DTLS (Datagram TLS)
Multicast No Yes

CoAP Message Types

MESSAGE TYPES
  • Confirmable (CON): Requires acknowledgment from receiver (like TCP)
  • Non-confirmable (NON): Fire-and-forget, no acknowledgment
  • Acknowledgment (ACK): Response to confirmable message
  • Reset (RST): Indicates message error or rejection

CoAP Resource URIs

Similar to HTTP but designed for constrained devices:

coap://192.168.1.100/sensors/temperature
coap://[fe80::1]/actuators/led
coap://sensor.local/light/brightness

Example Requests:
- GET /sensors/temperature - Read temperature
- POST /actuators/led - Change LED state
- PUT /config/sampling - Update configuration
- DELETE /sensors/temp/log - Clear log

Java CoAP Client (using Californium)

import org.eclipse.californium.core.*;
import org.eclipse.californium.core.coap.*;
import org.eclipse.californium.core.coap.CoAP.ResponseCode;

// IoT device sending data via CoAP
public class IotCoapClient {

    private static final String SERVER_URI = "coap://192.168.1.100:5683";

    public void sendTemperature(double temp) {
        // Create CoAP client
        CoapClient client = new CoapClient(SERVER_URI + "/sensors/temperature");

        // Create payload
        String payload = String.format("{\"temp\":%.2f}", temp);

        // Send POST request (NON-confirmable for efficiency)
        CoapResponse response = client.post(
            payload,
            MediaTypeRegistry.APPLICATION_JSON,
            CoAP.Type.NON  // Non-confirmable
        );

        if (response != null) {
            System.out.println("Response Code: " + response.getCode());
            System.out.println("Response: " + response.getResponseText());
        } else {
            System.out.println("No response received (NON-confirmable)");
        }

        client.shutdown();
    }

    public void getConfiguration() {
        CoapClient client = new CoapClient(SERVER_URI + "/config");

        CoapResponse response = client.get();

        if (response != null && response.isSuccess()) {
            System.out.println("Config: " + response.getResponseText());
        }

        client.shutdown();
    }
}
USE COAP WHEN:
• Device has very limited RAM/CPU
• Network is unreliable with high packet loss
• Battery power is extremely constrained
• Multicast communication is needed (e.g., turn off all lights)
• Very low message overhead is critical

WEBSOCKETS FOR REAL-TIME IOT

What are WebSockets?

WebSockets provide full-duplex, bidirectional communication over a single TCP connection. Once established, both client and server can send messages at any time without the overhead of HTTP request-response cycles.

When to Use WebSockets in IoT

IDEAL USE CASES
  • Real-time dashboards: Display live sensor data to users
  • Remote control: Instant feedback when controlling IoT devices
  • Live monitoring: Stream video or audio from IoT cameras
  • Collaborative IoT: Multiple users viewing the same data
  • Instant alerts: Push alerts to users immediately

WebSocket vs HTTP Polling

HTTP Polling (Inefficient):
Client: GET /data ← (every second)
Server: {"temp": 23.5}
Client: GET /data ← (every second)
Server: {"temp": 23.7}
Client: GET /data ← (every second)
Server: {"temp": 23.6}
... lots of HTTP overhead, even if data hasn't changed

WebSocket (Efficient):
Client: WebSocket Upgrade Request
Server: WebSocket Connection Established ✨

Server: {"temp": 23.5} ← (only when data changes)
Server: {"temp": 23.7} ← (only when data changes)
Server: {"temp": 23.6} ← (only when data changes)
... instant, bidirectional, low overhead

Java WebSocket Client

import javax.websocket.*;
import java.net.URI;

// WebSocket client for real-time IoT data
@ClientEndpoint
public class IotWebSocketClient {

    private Session session;
    private WebSocketContainer container;

    public void connect(String serverUri) throws Exception {
        container = ContainerProvider.getWebSocketContainer();
        URI uri = URI.create(serverUri);

        session = container.connectToServer(this, uri);
        System.out.println("Connected to WebSocket server");
    }

    @OnMessage
    public void onMessage(String message) {
        System.out.println("Received: " + message);

        // Parse and handle IoT data
        if (message.contains("temperature")) {
            updateTemperatureDisplay(message);
        } else if (message.contains("alert")) {
            handleAlert(message);
        }
    }

    @OnOpen
    public void onOpen(Session session) {
        System.out.println("WebSocket session opened");
        this.session = session;

        // Subscribe to specific topics
        sendMessage("{\"topic\":\"subscribe\", \"channels\":[\"sensors\",\"alerts\"]}");
    }

    @OnClose
    public void onClose(Session session, CloseReason reason) {
        System.out.println("WebSocket closed: " + reason);
    }

    @OnError
    public void onError(Session session, Throwable error) {
        System.err.println("WebSocket error: " + error.getMessage());
    }

    public void sendMessage(String message) {
        if (session != null && session.isOpen()) {
            session.getAsyncRemote().sendText(message);
        }
    }

    public void sendCommand(String deviceId, String command) {
        String json = String.format(
            "{\"deviceId\":\"%s\",\"command\":\"%s\",\"ts\":\"%s\"}",
            deviceId,
            command,
            java.time.Instant.now().toString()
        );
        sendMessage(json);
    }

    public void disconnect() throws Exception {
        if (session != null && session.isOpen()) {
            session.close();
        }
    }

    private void updateTemperatureDisplay(String message) {
        // Update UI with new temperature
    }

    private void handleAlert(String message) {
        // Handle alert - show notification, etc.
    }

    public static void main(String[] args) {
        IotWebSocketClient client = new IotWebSocketClient();
        try {
            client.connect("ws://iot-dashboard.example.com:8080/iot");

            // Send occasional commands
            while (true) {
                // Example: turn on fan when temperature is high
                client.sendCommand("actuator-001", "status");
                Thread.sleep(10000);
            }
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            try {
                client.disconnect();
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }
}

PROTOCOL COMPARISON & SELECTION

Decision Matrix

Scenario Best Protocol Why
Sensor data to cloud API HTTP or MQTT HTTP for simplicity, MQTT for scale
1000+ sensors to single gateway MQTT Publish/subscribe scales well
Limited battery, unreliable network CoAP UDP-based, minimal overhead
Real-time dashboard for users WebSockets Instant bidirectional updates
Mobile app controlling lights MQTT or WebSockets MQTT for IoT, WS for web apps
Industrial PLC communication HTTP or MQTT Depends on existing infrastructure

Performance Characteristics

BANDWIDTH OVERHEAD
  • CoAP: ~4-20 bytes per message (smallest)
  • MQTT: ~2-20 bytes header + topic
  • HTTP: ~500+ bytes per request/response
  • WebSockets: ~2-14 bytes after connection
LATENCY
  • CoAP: 1-5 ms (single packet)
  • MQTT: 5-50 ms (depends on QoS)
  • HTTP: 50-500 ms (TCP handshake included)
  • WebSockets: 1-5 ms (after connection)
POWER CONSUMPTION
  • Most Efficient: CoAP (UDP, small packets)
  • Very Efficient: MQTT (lightweight)
  • Moderate: WebSockets (persistent connection)
  • Less Efficient: HTTP (Frequent handshakes)

Protocol Selection Flowchart

Start
  ↓
Is device extremely constrained (low RAM/CPU)?
  ↓ Yes → Use CoAP
  ↓ No
Need real-time dashboard or user interface?
  ↓ Yes → Use WebSockets
  ↓ No
Have many devices (100+)? Need publish/subscribe?
  ↓ Yes → Use MQTT
  ↓ No
Is web API integration required?
  ↓ Yes → Use HTTP
  ↓ No
Are there existing web services infrastructure?
  ↓ Yes → Use HTTP
  ↓ No → Consider MQTT for flexibility

In practice, many systems use MULTIPLE protocols:
- Sensors → MQTT → Gateway → HTTP → Cloud API
- Mobile apps → WebSockets ⇒ Dashboard
- Ultra-low power beacons → CoAP → Collector
PRACTICAL IOT ARCHITECTURE:
Most real-world IoT systems use a hybrid approach:
1. Edge devices use MQTT or CoAP
2. Gateway/translates to HTTP/REST for cloud APIs
3. Web clients use WebSockets for real-time updates
Each protocol serves its purpose in the system!
