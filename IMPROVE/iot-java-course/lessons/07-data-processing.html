IoT Data Management Overview

From Sensor to Insight: Handling IoT Data at Scale

IoT systems generate massive volumes of time-series data continuously. Effectively processing, storing, and analyzing this data is critical to extracting value. This module covers the data pipeline from devices through processing to storage and analytics.

IOT DATA CHARACTERISTICS
  • High Velocity: Continuous streams from thousands of devices
  • Time-Series: Timestamped measurements over time
  • Volume: Petabytes across deployments
  • Variety: Structured, semi-structured, and unstructured data
  • Value Density: Often sparse; anomalies carry value

IoT Data Architecture

TYPICAL IOT DATA PIPELINE
  • Devices: Sensors generate raw measurements
  • Gateways: Aggregate, filter, initial processing
  • Ingestion: MQTT brokers, message queues
  • Stream Processing: Kafka, Flink, Spark Streaming
  • Time-Series DB: InfluxDB, TimescaleDB, Prometheus
  • Data Lake: S3, Azure Blob Storage for analytics
  • Analytics: Dashboards, ML models, alerting
EXAMPLE: Data Flow for Smart Building

Sensors (500 devices) → MQTT Broker → Kafka Stream Processor → InfluxDB (real-time) + S3 (archived) → Grafana Dashboard + Alerts

Time-Series Databases

Optimized Storage for IoT Sensor Data

Time-series databases (TSDBs) are purpose-built for storing timestamped sequence data points. Traditional SQL databases struggle with IoT workloads due to ingestion rate and query pattern requirements. TSDBs provide: high write throughput, efficient time-based queries, automatic downsampling, and retention policies.

POPULAR TIME-SERIES DATABASES
  • InfluxDB: Purpose-built TSDB, Flux query language, high performance
  • TimescaleDB: PostgreSQL extension, familiar SQL ecosystem
  • Prometheus: Monitoring-focused, Pull-based metrics, alerting
  • OpenTSDB: Hadoop-backed, scales to petabytes
  • QuestDB: SQL-compatible, high throughput, PostgreSQL protocol

InfluxDB with Java

WRITING IOT DATA TO INFLUXDB
import influxdbclient.InfluxDBClient;
import influxdbclient.WriteApi;
import influxdbclient.domain.WritePrecision;

public class IoTDataWriter {
    private InfluxDBClient client;
    private WriteApi writeApi;
    
    public void connect(String url, String token, String bucket, String org) {
        client = InfluxDBClientFactory.create(url, token.toCharArray());
        writeApi = client.getWriteApi();
    }
    
    public void writeSensorReading(String deviceId, double temperature, double humidity) {
        Point point = Point.measurement("sensor_reading")
            .addTag("device_id", deviceId)
            .addTag("type", "DHT22")
            .addField("temperature", temperature)
            .addField("humidity", humidity)
            .time(System.currentTimeMillis(), WritePrecision.MS);
            
        writeApi.writePoint(bucket, org, point);
    }
    
    public void close() {
        client.close();
    }
}

Querying Time-Series Data

FLUX QUERY ANALYSIS
// Flux query: Temperature aggregation by hour
from(bucket: "iot_data")
    |> range(start: -24h)
    |> filter(fn: (r) => r._measurement == "sensor_reading")
    |> filter(fn: (r) => r._field == "temperature")
    |> aggregateWindow(
        every: 1h,
        fn: mean,
        createEmpty: false
    )
    |> yield(name: "hourly_avg")

// Flux query: Anomaly detection (values > 3σ from mean)
from(bucket: "iot_data")
    |> range(start: -7d)
    |> filter(fn: (r) => r._measurement == "sensor_reading")
    |> filter(fn: (r) => r._field == "temperature")
    |> stddev() |> yield(name: "stddev")
    |> mean() |> yield(name: "mean")
DOWNSAMPLING & RETENTION POLICIES

Raw IoT data grows rapidly. Implement downsampling (e.g., keep raw for 7 days, 5-minute averages for 90 days, hourly averages for 1 year, daily averages forever). Retention policies automatically delete old data to control storage costs.

        ObjectMapper mapper = new ObjectMapper();
        String json = mapper.writeValueAsString(telemetry);

            System.out.println("Message sent successfully");
        }
    }
}

Stream Processing

Real-Time Data Processing with Kafka and Flink

Stream processing enables real-time analysis of continuous data flow. Unlike batch processing which processes data at fixed intervals, stream processing reacts to data as it arrives, enabling immediate alerts, aggregations, and transformations.

STREAM PROCESSING CONCEPTS
  • Event Time vs Processing Time: When event occurred vs when processed
  • Windowing: Grouping events by time (tumbling, sliding, session)
  • State Management: Maintaining context across events (counters, averages)
  • Exactly-Once Semantics: Processing each event exactly once
  • Watermarks: Handling late-arriving data

Kafka Streams with Java

KAFKA STREAMS: SENSOR AGGREGATION
import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;

public class SensorAggregationStreams {
    public KafkaStreams createStream() {
        Properties props = new Properties();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "iot-sensor-agg");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
        
        StreamsBuilder builder = new StreamsBuilder();
        
        KStream<String, SensorReading> readings = builder.stream(
            "sensor-readings",
            Consumed.with(Serdes.String(), new SensorReadingSerde())
        );
        
        // Windowed aggregation: 1-minute tumbling windows
        KTable<Windowed<String>, SensorStats> stats = readings
            .groupByKey()
            .windowedBy(TimeWindows.of(Duration.ofMinutes(1)))
            .aggregate(
                SensorStats::new,
                (key, reading, stats) -> stats.add(reading),
                Materialized.with(Serdes.String(), new SensorStatsSerde())
            );
        
        // Emit anomalies (temperature > threshold)
        readings.filter((key, reading) -> reading.getTemperature() > 50)
            .to("alerts");
            
        return new KafkaStreams(builder.build(), props);
    }
}

class SensorReading {
    private String deviceId;
    private double temperature;
    private double humidity;
    private long timestamp;
}

class SensorStats {
    private double avgTemp;
    private double maxTemp;
    private double minTemp;
    private int count;
}

Apache Flink CEP for Pattern Detection

FLINK CEP: FAILURE PATTERN DETECTION
import org.apache.flink.cep.*;

public class FailurePatternDetection {
    public Pattern<SensorEvent, ?> getFailurePattern() {
        return Pattern.<SensorEvent>begin("highTempStart")
            .where(event -> event.getTemperature() > 60)
            .next("highTempContinue")
            .where(event -> event.getTemperature() > 60, Times.secondsOf(60))
            .oneOrMore()
            .greedy()
            .until(event -> event.getTemperature() > 80);
    }
    
    public void detectFailures(DataStream<SensorEvent> events) {
        PatternStream<SensorEvent> pattern = CEP.pattern(
            events.keyBy(SensorEvent::getDeviceId),
            getFailurePattern()
        );
        
        DataStream<Alert> alerts = pattern.select((patternMap) -> {
            long duration = calculateDuration(patternMap);
            return new Alert("CRITICAL", 
                "Device overheating for " + duration + " seconds");
        });
        
        alerts.addSink(new AlertSink());
    }
}
EXAMPLE: Window Types
  • Tumbling Window: Fixed-size, non-overlapping (e.g., every 5 minutes)
  • Sliding Window: Fixed-size with slide interval (e.g., 5-minute window every 1 minute)
  • Session Window: Dynamic based on inactivity gap (e.g., user sessions)
  • Global Window: All events (with triggers for processing)

Edge Computing

Processing Data Closer to Devices

Edge computing moves data processing from centralized clouds to locations closer to data sources. This reduces latency, bandwidth usage, and enables operation during connectivity outages. Edge processing ranges from lightweight filtering on gateways to full application hosting on edge servers.

EDGE COMPUTING BENEFITS
  • Reduced Latency: Processing in milliseconds vs seconds
  • Bandwidth Savings: Send only processed insights to cloud
  • Offline Operation: Continue function during network outage
  • Privacy: Keep sensitive data on-premises
  • Cost: Reduce cloud compute and storage costs

Edge Continuum

EDGE LAYERS
  • Device Edge: Sensors and actuators, minimal compute
  • Gateway Edge: Protocol translation, aggregation, basic processing
  • Near Edge: Factory/building servers, localized apps
  • Far Edge: Regional data centers, CDN edge locations
  • Cloud: Centralized processing, analytics, ML training

Java Edge with Eclipse Kura

EDGE IOT APPLICATION WITH KURA
import org.eclipse.kura.KuraException;
import org.eclipse.kura.cloud.CloudClient;
import org.eclipse.kura.configuration.ConfigurableComponent;
import org.eclipse.kura.position.Position;
import org.eclipse.kura.timer.Timer;

public class EdgeDataProcessor implements ConfigurableComponent {
    private CloudClient cloudClient;
    private Timer processingTimer;
    private CircularBuffer<SensorData> buffer;
    
    @Activate
    public void activate(Map<String, Object> properties) {
        // Initialize edge processing
        buffer = new CircularBuffer<>(1000);
        processingTimer = new Timer(this::processBatch, 5000); // Every 5s
        
        System.out.println("Edge processor activated");
    }
    
    @Deactivate
    public void deactivate() {
        processingTimer.disable();
    }
    
    @SuppressWarnings("unchecked")
    @Reference
    public void setCloudClient(CloudClient cloudClient) {
        this.cloudClient = cloudClient;
    }
    
    public void onSensorData(SensorData data) {
        // Filter noise at edge
        if (isValidData(data)) {
            buffer.add(data);
            
            // Immediate alert if critical
            if (data.getTemperature() > 80) {
                sendAlert("CRITICAL_TEMP", data);
            }
        }
    }
    
    private void processBatch() {
        // Aggregate buffered data
        SensorAggregation agg = aggregate(buffer);
        
        // Send only aggregated result to cloud (90% bandwidth reduction)
        cloudClient.publish(agg.toJson(), "sensor/aggregated", 
            QoS.AT_LEAST_ONCE, true);
            
        buffer.clear();
    }
    
    private SensorAggregation aggregate(CircularBuffer<SensorData> buffer) {
        double min = Double.MAX_VALUE;
        double max = Double.MIN_VALUE;
        double sum = 0;
        int count = 0;
        
        for (SensorData data : buffer) {
            min = Math.min(min, data.getTemperature());
            max = Math.max(max, data.getTemperature());
            sum += data.getTemperature();
            count++;
        }
        
        return new SensorAggregation(
            min, max, sum / count, count, System.currentTimeMillis()
        );
    }
}
EXAMPLE: Edge vs Cloud Processing Split

Edge Processing: Real-time alerts, data filtering, deduplication, compression.

Cloud Processing: Historical analytics, ML model training, cross-site aggregation, reporting.

EDGE DEPLOYMENT CHALLENGES
  • Edge devices have limited CPU/memory and may be offline
  • Deployment requires robust OTA (Over-The-Air) update system
  • Edge and cloud codebases must sync when possible
  • Monitoring at edge requires careful design for intermittent connectivity
